走進任何一個用 LLM 架構，生產級 agent 系統的團隊，你會聽到幾乎同一套劇本：我們每次使用者請求平均要呼叫模型 5 到 15 次，但大部分的呼叫都是短而結構化的——查資料、做比對、 submit 一個 decision。多數團隊的預設做法是：全部 routing 到同一套旗艦模型，無論那個步驟的認知難度有多低。

這個預設從來沒有被嚴格驗證過。現有的工具呼叫 benchmark——BFCL、API-Bank、Gorilla——都是單輪評測，無法測出生產系統中那種「一個步驟依賴上一個步驟輸出」的相依性。ToolBench、WebArena、SWE-Bench 這些多步驟 suite 離真實場景更近，但它們纏繞了太多 confounds：API 漂移、web 與 GUI 接地、可能的 contamination。沒有人用一個乾淨、受控的環境，把工具使用這件事從「它到底在哪個認知層次失靈」這個問題裡剝離出來。

Harvard 的 Ranit Karmakar 與 Jayita Chatterjee 設計了 AgentFloor，試圖回答一個非常具體的問題：**小型開放模型（open-weight）究竟能在工具使用階梯上爬到多高？**

---

## 讓問題現形：為什麼舊評測回答不了這題

在深入方法之前，先把為什麼現有 benchmark 解決不了這個問題的核心原因說清楚。

現有 agent 評測試圖在同一套試卷裡測量太多東西。一個 WebArena 的分數，同時夾雜了：視覺 grounding 的能力、真實 API 的穩定性（ToolBench 在重跑時只有 44.4% API 成功率）、以及試題在預訓練語料中已被汙染的可能性。SWEBench Verified 在 2025 年重新分析後發現 59.4% 的audit 問題有缺陷的測試案例。這些 confounds 讓你根本無法說：「這個分數低，是因為模型工具使用能力不足，還是因為它在一個測試環境上卡在 API drift？」

更根本的問題是：多步 agent 任務通常是一次測所有能力——你不會知道失敗究竟發生在哪一個認知環節。是模型不會規劃？是它選錯工具？還是它壓根沒有持續追蹤約束條件的能力？

這些問題為什麼重要？因為如果你要設計一個 production routing 系統，你需要的不是一個 aggregate 數字，你需要的是一張**能力階梯圖**：模型在哪個層次還可以，在哪個層次開始明顯失敗。只有那張圖，才能讓你做出「這步用小模型、那步才調旗艦」的決定。

---

## AgentFloor 的設計：乾淨的六層階梯

AgentFloor 的核心設計是：把工具使用的認知需求，拆成六層逐步增加的難度階梯，每層五個任務，共 30 題。

| 層次 | 認知需求 | 步驟上限 |
|------|----------|----------|
| A0 | 無工具的指令遵循 | 1 |
| A | 單一工具呼叫 | 2 |
| B | 順序兩步工具鏈（輸出餵給輸入）| 4 |
| C | 條件分支（根據中間結果）| 6 |
| D | 多源綜合與衝突還原 | 8 |
| E | 長期規劃與持久約束追蹤 | 10 |

所有任務都走同一組八個抽象工具（search_records、lookup_record、get_attribute、list_options、check_constraint、compare_records、compute_value、submit_decision），對接一個 in-memory fixture database。沒有檔案系統、沒有外部服務、沒有時間變動狀態，也沒有可能被預訓練語料汙染的路徑。

這個設計的關鍵好處是：**把環境噪聲全部拿掉，讓認知需求的差異完全來自任務本身的邏輯結構**，而不是 API 漂移或 contamination。

研究團隊測了 16 個 open-weight 模型，從 0.27B 到 32B 參數， Alongside GPT-5 family，透過 Ollama 跑本地推論，用 OpenAI Responses API 跑 GPT-5，做了 16,542 次評分 run。

## 答案比直覺更複雜：一個「並非均勻」的畫面

最粗粒度的結果在 aggregate 層次：gemma4:26b 與 GPT-5 在全部 30 題上打成平手，Δ=+0.4 百分點，90% CI [−4.0, +5.1]，落在 pre-registered 的 ±10pp 等價界線內。兩者的總 TCR 都是大約 60%，但 gemma4:26b 在 Mac Studio 上每個通過任務只要 $0.0022，GPT-5 要 $0.0327——差了 15 倍。雲端 GPU 場景下，大約是 3 倍差距。

這個 aggregate parity 聽起來像是「小型 open-weight 模型已經能完全取代 GPT-5」。但這是一個危險的詮釋。當你把結果切到 tier 層次，畫面完全不同。

**A0 層（無工具指令遵循）：gemma4:26b 100% 對 GPT-5 80%，小模型顯著勝出。** 有趣的是，GPT-5 在這層的唯一失敗題型是同一題（A02，任務是從事故摘要中抓出要素並列點），失敗模式是 F5：模型用自己的話重新詮釋了事實，使用的技術術語與 keyword matcher 無法對上，導致評測器判定失敗。小模型反而靠「忠實複製」拿到全分。

**A 層（單一工具呼叫）：兩者等價，Δ=+2.2pp，95% CI 完全落在 ±10pp 內。**

**B 層（順序工具鏈）、C 層（條件分支）、D 層（多源綜合）：點估計差距都在 9pp 以內，但信賴區間太寬（每層 n=45，95% CI 寬度 24–35pp），不足以在 ±10pp 界線下宣稱等價。** 數據沒有足夠統計力度，無法確認也無法否認。這不是一個「小模型不行」的故事，而是一個「樣本不夠大，無法給出結論」的故事。

**E 層（長期規劃與持久約束追蹤）：GPT-5 明確勝出。** gemma4:26b 0% 對 GPT-5 10.2%，Δ=−10.2pp，95% CI [−18.4, −2.0]，嚴格顯著。在這個層次，兩者都還沒有達到「可用於部署」的可靠度，但 GPT-5 的相對優勢是存在的。

## 這不是一個純粹的 scale 問題

E 層的 gap 最明顯，但研究團隊在 ablation 實驗中發現了一件麻煩的事：**這個 gap 無法用單一 universal 干預修補**。

他們測了幾種方法：結構化提示（要求模型 explicit 分步寫出規劃）、Qwen3 reasoning toggle、GPT-5 的 step budget 加倍到 20 步。其中一個看起來很直覺的做法——明確要求模型分步寫出執行計畫——實際上讓所有模型在所有層次的表現都變得更差。

更重要的發現是：干預的效果是模型specific的，不是 universal 的。對某個模型有效的干預，對另一個模型可能是 neutral 或負向的。這說明 E 層的失敗不是一個簡單的能力不足，而是不同模型在不同方向上以不同方式失敗。有些在約束追蹤上崩潰，有些在協調多個目標時放棄，有些在冗長的決策樹中迷路。

這個發現對 routing 的實際影響是：你不能簡單地說「E 層全部 routing 到 GPT-5」就能解決問題，因為即使在 GPT-5 這邊，E 層的絕對 pass rate 也只有 10%。你需要的可能是更細緻的任務特化策略——但那已經超出本論文直接回答的範圍。

## 成本與能力的真實交換：你在哪個層次省錢？

把數字放進實際部署的脈絡裡，會看到一個清楚的成本—能力 map：

- **A0、A 層（指令遵循、單工具）：可以果斷 routing 到 3B–8B 的 open-weight 模型**，不是邊緣接近，而是有統計強度支撑的等價。gemma4:26b、mistral-small3.2:24b、nemotron-3-nano:4b、qwen3:14b、qwen3:8b 這五個模型在 A0 層 Holm-pass 了非劣性測試。這些層次佔了大部分生產 agent 流程中的呼叫量。

- **B、C、D 層（鏈接、條件、多源）：沒有結論，但也沒有噩耗**。點估計傾向接近 GPT-5，但信心不夠。這意味著在這些層次，你可以根據延遲與成本偏好做決定，不需要因為「數據說小模型不行」而放棄。

- **E 層（長期規劃）：仍然值得 routing 到前沿模型**，但要對絕對可靠度保持警覺。這一層的問題是：即使 GPT-5 也只有 10% 的 pass rate。這個層次本身還沒有被任何模型解決到「可部署」的標準。

## 為什麼這個研究值得認真看待

多數 agent 能力研究報告 aggregate 分數，然後說「模型 X 逼近 GPT-5」，但 агрегат 分數掩蓋了能力分布的異質性。一個 60% aggregate TCR 的模型，可能在某些層次是 100%，在另一些層次是 0%。如果你只是看 aggregate 數字就把所有任務 routing 到同一個模型，你同時做了兩個錯誤：讓簡單任務付出過多成本，以及在高難度任務上給了模型它其實沒有能力的錯覺。

AgentFloor 的貢獻在於它強迫你用 tier 視角做 routing 決定，而不是用 aggregate 視角。這也是為什麼它對 FrugalGPT、RouteLLM、Hybrid LLM 這類 routing 系統有意義——它們需要一個靜態的 prior 來指導「什麼難度的任務值得調用貴模型」。AgentFloor 提供了那張 prior map。

另一個值得注意的細節是研究團隊怎麼處理失敗模式分類。他們設計了一個嚴格的 priority cascade（F1 幻覺工具、F2 格式錯誤、F4 步數用盡、F5 提前放棄、F5b 從未啟動、F6 選錯工具、F7 部分完成、F3 殘差），F5 與 F5b 的區分特別關鍵——它把「曾經成功執行過工具但最終放棄」和「根本沒有啟動」這兩種失敗分開，而這兩種失敗指向的修復方向截然不同。

## 一個部署工程師會帶走的結論

如果你正在架構一個 production agent 系統，AgentFloor 的結果告訴你一件事很清楚：**把 agent 工作流切成能力層次，是值得的**。

不是所有模型呼叫都生來平等。當你的系統需要的是 A 層的「查一筆 record 並取出屬性」，讓一個 3B 的 local 模型處理，延遲更低、成本只有 GPT-5 的幾十分之一，而你在 same-tier 上沒有 measure 到的 loss。當你的系統需要跨十步協調與持久約束追蹤，你仍然應該讓前沿模型處理，但同時也要對它的極限保持現實——它也不過是 10% 的 reliable。

這個研究所不能告訴你的是：如何在真實部署中自動化這個分層決策。因為 B/C/D 層的信心不足是統計問題，不是能力問題——更大的 paired sample 可能會收斂出清楚的結論。但如果你正在設計一個靜態的 routing ruleset，這張 map 已經足夠你做出有意義的成本節省決定。

不是所有工具都需要最大的模型。這句話看起來直覺，但它的代價是：我們大多數人從來沒有真的檢驗過它。
