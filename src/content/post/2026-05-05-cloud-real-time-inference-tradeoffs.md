## 大家一直把同一筆帳算錯地方

在做 edge AI 的系統設計時，幾乎每個人都聽過同一句話：只要牽涉即時控制，推論就該盡量留在裝置本地。理由很直覺——雲端再快，也得先吃一段網路延遲；車子要煞車、機器手臂要停下來、工廠相機要即時判斷瑕疵，哪有空等封包跑來跑去。

這篇 arXiv 論文《Cloud Is Closer Than It Appears》想挑戰的，就是這個幾乎沒人重算過的前提。作者把問題拆得工程：如果真正決定安全性的，不是「有沒有網路」這一件事，而是整條 perception-to-action 鏈路的總反應時間，那麼我們平常把 cloud 先判死刑，可能只是因為只盯著網路這一小段。

他們挑了一個容易讓人緊張的場景來驗證：自駕車的緊急煞車。感測器拍到前方障礙物後，影像要進模型，模型要完成偵測，控制系統再發出煞車指令。這整條鏈只要慢一點，留給車子的減速距離就會變少。

## 真正該比的，不是網路延遲，而是整體反應預算

論文的核心判斷其實很乾脆：**只要雲端往返延遲，小於端上與雲端之間省下來的推論時間差，雲端就可能更快。**

這句話聽起來像常識，但大家平常少這樣算。多數部署討論只看到「端上沒有 network penalty」，卻沒把另外幾筆帳放進來：裝置上的模型通常得選得更小、推論吞吐比較差、排隊延遲更容易累積，還得受電力與散熱預算限制。雲端則相反，網路是額外成本，但 GPU 吞吐高得多，還能撐更大的模型。

作者把這件事形式化成一個 latency model，變數包括感測頻率、平台吞吐、網路延遲與安全約束。論文裡最有力的一條結論是 Lemma 1：若 cloud 的傳輸延遲，小於兩端平台在「排隊攤平後」的推論延遲差，cloud response latency 就會低於 device。這個觀點逼你承認，部署位置不是信仰，而是 end-to-end budget。

## 當更大的模型碰上更快的 GPU，22ms 網路不一定輸

作者把理論放進 CARLA 模擬器裡做緊急煞車實驗，感測模型採用 Ultralytics 的 YOLO11 系列。端上平台是 Jetson AGX Orin，雲端平台則是一台專用 RTX A5000 GPU server。為了不要把結論建立在虛構網路上，他們還用 AWS EC2 Local Zone 測了實際 round-trip latency，再用 Wi‑Fi 與 5G 的 p10、p50、p90 延遲分佈去模擬網路條件。

實驗配置裡有兩個數字很關鍵。第一，論文的 Table 1 指出，A5000 的推論延遲大約比 Jetson 低 **85%**。第二，根據 YOLO11 官方文件，YOLO11m 的 COCO detection mAP 是 **51.5**，YOLO11x 則是 **54.7**；也就是說，雲端不只比較快，還能承載更大的模型。論文選擇 device 端部署 YOLO11m，cloud 端部署 YOLO11x，前提是兩者都必須滿足 **100ms** 的控制延遲約束。

結果很反直覺。文中描述，在 **22ms** 的中位數 round-trip delay 下，雲端方案仍然能更早發出煞車，而且平均能比端上**提早約 20 公尺**偵測到障礙。原因不是單一因素，而是兩件事一起發生：雲端 GPU 把推論時間壓得夠低，較大的 YOLO11x 又能更早看出前方物體。當車速提高時，這個差距更明顯，因為留給系統反應的時間本來就已經被壓縮。

我喜歡這個結果的地方，是它沒有把 cloud 神化成萬靈丹。作者自己也承認，他們用的是 dedicated RTX A5000 server，不是會受多租戶干擾的公有雲共享節點。換句話說，論文證明的不是「雲端永遠贏」，而是「只要基礎設施夠近、算力夠密，cloud-first 在即時場景裡不是禁忌」。
## 這篇論文真正動到的，是 edge-first 的默認值

對工程師來說，這篇文章最重要的收穫不是「把所有東西搬去雲上」，而是部署討論該換一套問法。你該問的不是「控制系統能不能容忍網路」，而是「在相同安全約束下，哪個部署位置能給我更好的總反應時間、模型精度與能耗平衡」。這三個量不該分開看。

這也讓 AWS Local Zones 這類基礎設施變得比較有意思。官方把它定位成把雲端服務拉到都會區、支援 latency-sensitive applications 的中介層；放進這篇論文的框架後，它開始變成可以量化的架構選項。

如果要挑這篇論文可能被高估的地方，我會放在外部條件上：local zone 並不是每個地區都有，專用 GPU 也不是每個團隊都租得起，真實部署還得面對網路抖動、可用區成本、容錯路徑與資料治理。可它被低估的地方同樣明顯——它逼大家重新承認一件事：在 2026 這種基礎設施條件下，「端上一定比較安全」已經不是可以直接照抄的結論了。

有些系統依然會留在 edge，這很正常。但如果你今天還用五年前那套直覺決定 inference placement，恐怕不是你比較保守，而是你的 latency 帳本還沒算完。
