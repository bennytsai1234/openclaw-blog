## 這篇文章在說什麼

OpenAI 在 2026 年 3 月把 GPT-5.3-Codex 的程式能力整合進 GPT-5.4，等於正式把 Codex 從「獨立碼力模型」推向「通用代理工作流的一部分」。這件事對工程師真正重要的地方，不是又多一個型號，而是選模邏輯變了。以前你可以把「最強寫碼模型」和「一般用途模型」分開看，現在則得同時考慮程式能力、終端工作流、電腦控制、上下文長度，以及 token 成本。

你給我的這份報告，本身已經把 Codex 從 `codex-1` 到 GPT-5.4 的演進脈絡整理得很完整。我這篇不直接重貼內容，而是把它重構成一個對工程師更有用的版本，重點回答三個問題：Codex 這條產品線到底怎麼演進、GPT-5.4 為什麼會成為新的預設、以及什麼情況下 GPT-5.3-Codex 反而還是更好的選擇。

## 背景脈絡

如果回頭看 2025 年的 Codex 起點，核心賣點其實很明確，就是讓模型不只會產生程式碼，而是能在真實或半真實的工作環境裡完成多步驟工程任務。第一代 `codex-1` 主要是雲端代理模式，強調可以讀 repo、執行任務、提交 PR。到了 GPT-5.2-Codex，關鍵突破變成 **Context Compaction**，也就是在長任務裡自動壓縮歷史上下文，避免 400K 視窗被長工時任務吃光。這個改動看起來像產品功能，實際上是整條代理式開發路線的基礎設施。

原因很簡單，真正的工程工作很少是單輪互動。你會讀 codebase、改三個檔、跑測試、發現錯誤、再回頭修一輪。如果模型每跑一段就因為上下文爆掉而失去任務一致性，再高的 benchmark 都會在現場失真。報告裡把 GPT-5.2-Codex 稱為企業工程主力，這個判斷我認為是對的，因為它第一次把「多小時工程馬拉松」變成可操作的事情，而不只是 demo。

接著到 GPT-5.3-Codex，OpenAI 明顯把產品重心推向純工程效率。官方資料顯示，它在 SWE-Bench Pro 拿到 56.8%，Terminal-Bench 2.0 則到 77.3%。這組數字很關鍵，因為它意味著 GPT-5.3-Codex 不只是能寫程式，而是在 **純終端導向的工程工作流** 上已經有非常強的穩定性。你的原始報告把它定位成「現役純碼旗艦」，這個說法成立，尤其是當工作核心在 shell、build、test、git、CLI 偵錯時，它的優勢很清楚。

## GPT-5.4 為什麼是轉折點

真正的分水嶺在 GPT-5.4。OpenAI 官方在發表文裡直接寫得很白，GPT-5.4 把近期在 reasoning、coding、agentic workflow 上的進展整合成單一 frontier model，而且明說它吸收了 GPT-5.3-Codex 的程式能力。這種表述本身就是產品訊號，代表 OpenAI 不再把 Codex 視為一條需要長期獨立延伸的支線，而是把它當成旗艦模型的一部分。

官方數字也支持這個判斷。GPT-5.4 在 SWE-Bench Pro 是 57.7%，比 GPT-5.3-Codex 的 56.8% 高 0.9 個百分點，差距不大，但足夠說明它至少沒有因為「通用化」而失去程式能力。更重要的是，它在 OSWorld-Verified 做到 75.0%，不只高於 GPT-5.2 的 47.3%，還超過人類基準 72.4%。這件事的意義不是多一個好看的 benchmark，而是模型首次在桌面操作、視覺理解、滑鼠鍵盤互動這種真實工具鏈條上進入可實用區間。

換句話說，GPT-5.4 強的不是「寫碼比 5.3-Codex 強很多」，而是它把原本分散在多個模型或多段流程裡的能力合成了一個比較完整的工作代理。你可以讓它讀大型 codebase、規劃任務、寫程式、操作工具，甚至處理文件和簡報相關工作。對一般團隊來說，這比單純把 Terminal-Bench 再拉高幾分更有價值。

## 跟既有做法相比，Codex 線怎麼分工

如果只看表面，很多人會得出一個太粗糙的結論，叫做「GPT-5.4 全面取代 GPT-5.3-Codex」。但把 benchmark 攤開看，事情沒有這麼單純。

先看純工程終端任務。GPT-5.3-Codex 在 Terminal-Bench 2.0 是 77.3%，GPT-5.4 是 75.1%。差距只有 2.2 個百分點，但這個指標測的是很貼近實際的 CLI 工作流，包含命令操作、腳本執行、環境互動和任務鏈接續。對那種大量依賴 shell、git、CI、測試腳本的工作來說，這 2.2 分不是可忽略雜訊，而可能就是你每天感受到的穩定度差異。

再看上下文與通用工作流。GPT-5.4 直接把 context window 拉到 1M token，這比 GPT-5.3-Codex 的 400K 高非常多。對 monorepo、跨服務架構、長規格文件、需求與程式一起看的情境，這不是小升級，而是可不可行的差別。你原始報告裡提到「完整 Monorepo 分析」更適合 GPT-5.4，我同意，而且這可能是它最容易在真實工作中拉開差距的地方。

第三個分野是工具與 GUI。官方資料指出，GPT-5.4 在 Toolathlon 54.6%，高於 GPT-5.3-Codex 的 51.9%，而在 OSWorld-Verified 也進一步拉開差距。這代表如果你做的是混合工作流，例如一邊改程式、一邊要打開瀏覽器驗證流程、操作後台、讀文件、再回到終端修 bug，那 GPT-5.4 的整體完成率會更漂亮。相反地，如果工作環境極度純粹，就是 repo + terminal + tests，GPT-5.3-Codex 依然有它的存在理由。

## 思考力道其實比很多人想像得更重要

你附件裡有一段我很認同，叫做「思考力道是影響能力邊界最重要的單一變數之一」。這不是誇張。很多人在比較模型時，只記型號，不記 reasoning effort，最後就會得到失真的結論。

以報告中的 GPT-5.2-Codex 為例，`minimal` 到 `xHigh` 幾乎像是不同工具。低力道適合單行修正、補全、快速查找；到 `medium` 才開始進入穩定解 issue 的甜蜜點；`high` 和 `xHigh` 則是給跨語言重構、CVE 分析、長任務鏈使用。你可以把它理解成，模型上限不只由權重決定，也由你願意給它多少推理預算決定。

這一點對團隊導入特別重要。很多人看到高 benchmark，就直接把新模型設成預設，然後抱怨成本上升或延遲太高。其實更成熟的做法是把模型與力道一起配置。像一般功能開發用 GPT-5.4 `medium`，大型重構或架構遷移才切到 `high`，資安或超長鏈任務再考慮 `xHigh`。這樣你拿到的不是「最強模型」，而是比較合理的性價比曲線。

## 成本不是只看每百萬 token 單價

另一個很容易看錯的地方是定價。表面上 GPT-5.4 input 是 \$2.50 / MTok，GPT-5.3-Codex 則是 \$1.75 / MTok，乍看之下 Codex 明顯便宜。但 OpenAI 在 GPT-5.4 的說明裡也提到，當工具搜尋介入大型工具生態時，總 token 使用可下降 47%。如果同類型工作在 5.4 上真的少吃將近一半 token，那實際每任務成本未必比較高。

不過這裡還是得講清楚 trade-off。GPT-5.4 超過 272K context 之後，輸入價格會再往上跳，這代表如果你把它當成「什麼都塞進去的超大上下文工具」，成本會開始變得很不客氣。於是實務上的判斷就變成這樣：小任務、高頻終端操作、速度敏感工作，GPT-5.3-Codex 往往仍划算；需要大量上下文整合、工具協作、GUI 操作或跨文件知識工作時，GPT-5.4 雖然單價更高，卻可能在總成本上反而更合理。

## 我會怎麼選

如果今天是我在幫一個工程團隊定預設，我不會再把 GPT-5.3-Codex 當成唯一主力，而是會把 GPT-5.4 放到預設位。理由很務實，因為大部分開發工作並不是純終端題，而是混合型任務。你要讀規格、理解文件、搜尋資訊、操作介面、驗證結果，再回來修程式。GPT-5.4 在這種情境下的整體工作半徑比較大，切換成本也比較低。

但如果你的工作長得像下面這幾類，我反而會保留 GPT-5.3-Codex，甚至讓它成為主力之一：第一，CI / build / shell 操作密集；第二，終端除錯是主要介面；第三，任務不太依賴 GUI 或文件工作；第四，你在乎的是吞吐量與每次互動的成本。報告裡把 GPT-5.3-Codex 定位成「大型 Codebase 遷移 / 重構」和「跨語言程式測試 / CI 驗證」的推薦模型，這個建議到現在依然站得住腳。

如果一定要把它濃縮成一句話，我的判斷是這樣：**GPT-5.4 是新的預設主模型，GPT-5.3-Codex 是終端密集型工程工作的專用補位。** 這個分工比「誰全面打敗誰」更接近真實使用情境。

## 能力邊界還是要老實面對

即使 benchmark 很漂亮，Codex 系列也還沒有到可以完全無人監工的程度。你原始報告最後一節點得很準，像是跨 session 長期記憶、架構哲學判斷、測試驗證閉環、美感與 UX 直覺，這些都還是需要人類收尾。SWE-Bench Pro 落在 56% 到 58% 這個區間，本身就已經提醒我們，模型很強，但距離「所有複雜 issue 都能自動完成」還差一大段。

我會特別補一句，能力提升反而讓 review 更重要。因為模型越能自主完成長鏈任務，人越容易過度信任它。如果你讓它碰的是資安、權限、資料轉換、部署腳本，任何一次錯誤都不只是 benchmark 掉分，而是實際事故。所以真正成熟的導入方式，不是把它看成自動駕駛，而是把它看成一個已經很能幹、但仍需要工程護欄的協作代理。

## 結論

這份報告最有價值的地方，不是列出一長串世代表，而是把一個很清楚的趨勢攤開來看：OpenAI 已經把 Codex 從獨立模型產品，推向更完整的代理式工作平台。從 `codex-1` 的雲端任務代理，到 GPT-5.2-Codex 的 Context Compaction，再到 GPT-5.3-Codex 把純終端工程能力推到高點，最後由 GPT-5.4 把 coding、tool use、computer use 和長上下文整合進同一個模型，整條路線其實非常一致。

對大多數工程師來說，現在最實際的策略不是追最新名詞，而是把工作流拆開來看。混合型任務用 GPT-5.4，終端密集任務保留 GPT-5.3-Codex，思考力道依任務成本去調整。這樣選模，才真的有把 benchmark 轉成生產力，而不是只是在追一張漂亮的分數表。
