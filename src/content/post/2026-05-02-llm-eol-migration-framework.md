多數人在談 LLM 遷移的時候，腦中浮現的畫面是這樣的：某個模型即將停用，工程團隊趕在期限前測完幾個候選模型，挑一個分數最高的，然後上線。听起来很简单。

但如果你的系統每個月要處理 530 萬次互動，橫跨六個時區，支撐數十家企業客戶的客服情境，而整個遷移窗口只有幾個月——你會發現「分數最高的模型」這個答案根本不存在。因為自動化指標不可靠，人工評估數據又不夠多，而模型供應商停產模型的通知是不等人的。

Verint Systems 的團隊最近發表了一篇論文，記錄他們在這個場景下實際怎麼做的。核心不是發明新模型，而是一套**在資訊不足時依然能做出有信心決策**的框架。這個框架的基礎，是貝葉斯統計。

## 問題從來不是「有沒有新模型」

商業系統依賴第三方 LLM 的團隊，大概每隔一年就會遇到一次這樣的通知：某個型號即將從 Azure、AWS 或 GCP 下架。對於擁有十幾種 LLM 產品的企業來說，這不是單一產品的小更新，而是一次影響範圍橫跨整個產品矩陣的連鎖遷移。

你可能會問：那就直接換最新一代的模型就好了？但現實不是這樣運作的。

首先，模型不是向下相容的。同一套 prompt 在 Claude 3 Haiku 上可以穩定輸出正確的 XML 格式，到了 GPT-OSS 120B 就是會 random failure。格式控制只是基本門檻，接著還有正確率、拒答率、風格一致性、延遲、費用、區域可用性——每一個維度都是一組複雜的權衡。

其次，評估是一個非常貴的動作。理想情況下，你想找人對每個候選模型產出的回答做完整的人工評分，但 530 萬月互動的系統，問題類型分布極廣，而人工標註的速度根本追不上遷移窗口的壓力。公開的 QA 測試集（如 HotpotQA、SQuAD）數量足夠，但往往和企業實際客服情境脫節；自己建內部測試集，數據量又太小，導致結果的信心不足。

所以問題的本质不是「哪個模型更強」，而是：**在自動化指標不可信、人工資料又有限的情況下，如何有系統地比較模型，並對決策結果抱持合理信心。**

## Bayesian 作法：把不確定性放進模型裡

這篇論文最核心的方法論貢獻，是用 Bayesian 統計把「自動化指標的可信度」顯式建模出來。

具體做法是這樣的：先用一小批人工標註過的回答（他們實際標了 146 筆，分別來自 HotpotQA、內部測試集和 SQuAD），建立每個自動化指標的 confusion matrix。也就是說，我能知道「這個指標說這答案是對的，但人類事實上認為對不對」——進而算出這個指標的 true positive rate 和 false positive rate。

有了這些資訊之後，再用 Monte Carlo 方法模擬，在假設有更多資料的情況下，兩個模型之間的正確率差距落在哪個區間。這不是傳統的「點估計」，而是一個**信心區間**——它告訴你「以目前手上的資料，我們有 90% 的信心認為模型 A 比模型 B 領先 2% 到 8%」。這個範圍本身就已經是一個有價值的判斷，因為它同時量化了「差距是多少」和「這個差距有多確定」。

這個框架厲害的地方在於，人工評估只需要做一次，就能重複用於多個指標和多個模型的比較。他們測試了 RAGAS 正確率、faithfulness、relevance，以及兩個自訂指標，最後發現 `new_correctness`（拿完整來源上下文而非僅僅 gold-standard answer 作為評估基準）與人類判斷的擬合度最高，在兩個測試集上都達到了 90% 以上的 true positive rate。

## 先做排除法，再做選擇法

光有好的統計方法還不夠，還需要一套能實際運作的篩選流程。論文裡描述的六步框架值得拿出來仔細看，因為每一步背後都有明確的工程理由。

**Step 1** 是候選名單審查，過濾掉內部審核未通過（ bias、隱私、法規遵循、授權）、成本超標、或特定區域壓根沒有提供的模型。這一步跟模型能力無關，但直接影響最後能不能落地。

**Step 2** 是格式通過測試：把所有候選模型跑一遍簡單的格式任務（例如輸出指定格式的 XML），不能穩定通過的直接淘汰。他們在這一步就去掉了 OpenAI 兩個 GPT-OSS 模型，原因是無法穩定生成符合規範的 XML 標籤。這裡有一個細節：Qwen3-32B 一開始也有同樣問題，但後來發現只要在 prompt 裡加上幾個正確格式的範例就能解決，所以他們把這個當作快速修復通過了。

**Step 3** 是自動化指標的 Bayesian 校正，使用 `new_correctness` 評估每個候選模型在測試集上的表現，並同時追蹤 IDK（無法回答）率。這裡開始真正進入 Bayesian 的部分——不是只看分數，而是看分數區間。

**Step 4** 是門檻過濾：如果 IDK 率顯著上升、延遲超出範圍、或正確率相對於基準模型有統計顯著下降，直接淘汰。他們的說法是「有顯著增加」，這裡的「顯著」就是由 Bayesian 信心區間定義的，不是 p-value 判斷，而是「90% 信心區間的左端點是否為負」。

**Step 5** 是風格一致性檢查，確認回答中不出現「according to the sources」、「the knowledge provided」這類口語化或不正確的引用框架，以及回答長度是否合理。如果有風格問題，先嘗試調整 prompt，再重新評估正確率。這一步的順序很有講究——先確認模型能力底線，再處理風格表現，避免在能力不達標的模型上最佳化 prompt 白費力氣。

**Step 6** 是地域與功能覆蓋的最終選擇。如果單一模型無法覆蓋所有必要的區域和功能组合，就需要選擇多個模型，這時候才進入成本與品質的最優化階段。

## 數字背后的一些觀察

他們最後的候選池包括 Gemma 3 27B、Claude 4.5 Haiku、Claude 3.5 Sonnet、AWS Nova Micro/Lite/Lite 2/Pro，以及 Qwen3-32B 和 Qwen3-235B。最終選擇結果論文沒有完整揭露，但有幾個數字值得記錄。

在 HotpotQA 上，`new_correctness` 的 TPR（true positive rate）達到 0.883，遠高於 RAGAS 正確率指標的 0.750，以及 faithfulness 的 0.800。這印證了一件事：拿 gold-standard answer 作為唯一評估基準，在開放域問答上是有盲點的——因為 gold answer 常常不是唯一正確答案，嚴格匹配會錯誤懲罰那些同樣正確但表述不同的答案。

另外在 SQuAD 上的初步實驗讓他們決定放棄這個測試集，原因是內部測試集規模太小，導致 calibration 出來之後沒有足夠的 true negatives 做有效的指標校正。這是一個很重要的実務細節：不是所有測試集都適合 Bayesian calibration，要先看你的資料分佈是否支撐這個方法。

## 這個框架的真正價值

看完這篇論文，我的感受是：它解決的不是「選模型」這個問題，而是「在資訊不足的條件下如何有信心地選模型」的問題。

這兩件事看起來差不多，但對組織的影響完全不同。前者是技術決策，後者是組織決策。當一個模型遷移失敗（例如選了錯誤率比基準高 5% 的模型），受影響的不只是系統效能，還有客戶信任、內部對 AI 系統的信心，以及遷移決策的來龍去脈能否被回溯檢視。

Bayesian 方法在這裡的價值不是更準確的分數，而是**把不確定性顯示化**。一個 90% 信心區間 [2%, 8%] 的優勢，比一個單點估計「+5%」更有決策參考價值，因為前者告訴你「我們有 90% 的信心模型 A 比較好，但幅度可能在 2% 到 8% 之间」，而後者只是一個沒有上下文的神秘數字。

這種做法在統計文獻中並不新鮮，但在 LLM 工程實務中極為罕見。多數團隊的模型選型仍然是「跑幾個 benchmark，挑分數最高的」，而 benchmark 的分數從來不會告訴你「這個分數有多可靠」。這篇論文把這個gap指了出來，並且提供了一個可直接套用在實際產品系統上的解決方案。

## 框架的局限與適用邊界

當然，這不是一個可以完全不動腦筋就往任何系統上套的解決方案。論文本身也提到了幾個限制。

首先，人工標註雖然只需要一次，但前期建立標註規範的過程並不輕鬆。他們讓三個評估者共同討論並建立 guideline，光是marginal cases 的判斷原則就花了相當的時間。在一個小型團隊或時間壓力更大的情境下，這個成本可能高到讓人卻步。

其次， Bayesian calibration 的效果嚴重依賴於初始樣本的代表性。如果人工標註的範例恰好是模型擅長的類型，估計出來的 metric accuracy 就會偏樂觀，後續所有使用該指標做的決策都會受到影響。論文選擇隨機抽樣來緩解這個問題，但隨機抽樣在小樣本下本身也是一個不穩定的操作。

第三，這套框架目前只驗證在一個 QA 任務上。對於生成式摘要、程式碼生成、對話系統等其他任務類型，框架的各個步驟（格式測試門檻、IDK 率容忍度、風格檢查規則）都需要重新設計，不建議直接迁移。
