在遊戲裡打Boss，打了二十次還在同一個地方失敗——這種經驗，LLM 也有。一個用來當遊戲 agent 的大型語言模型，每次遇到類似的決策情境，反應可能都不一樣。它會推理、會规划，但有一件事它幾乎做不到：**把上一次學到的東西，帶到下一次去用**。

不是模型不夠大，也不是推理能力不夠強。是 LLM 壓根沒有一個機制，去「發現」「留住」和「取用」跨回合累積下來的結構化技能。你可以叫它記憶問題，也可以更精確地說——是一個**技能獲取與重用的結構性缺口**。

## 舊方法早就碰到牆，只是大家沒有明講

過去的遊戲 agent 無非幾條路：強化學習 train 死的策略網路、用 Rule-based 預先把技能寫死、或者乾脆靠 Prompt Engineering 把專家知識塞進模型。每一種都有代價。

強化學習出來的策略，能泛化但缺乏解釋性，而且每換一個遊戲幾乎要從頭 train 一遍。Rule-based 的好處是可預測，壞處是遇到沒寫到的情境就當機。把技能寫進 Prompt？看場景，也看模型上下文窗口還剩多少空間。

更深層的問題是：這些方法都預設**技能是從外面被給進去的**。模型不會自己從玩的過程裡長出技能，更不會把某一次推理成功的模式留給下次參考。每一局遊戲，對模型來說都是新的。

那，有沒有可能讓 LLM 一邊玩、一邊從自己的失敗軌跡裡長出技能庫？

## 真正的切入點：讓 agent 的「技能」自己長出來

COSPLAY 是 2026 年 4 月由馬里蘭大學、USC、MBZUAI 與 Good Start Labs 聯合發表的框架，全名是「Co-Evolving LLM Decision and Skill Bank Agents for Long-Horizon Tasks」。它的核心想法很直白：**與其從外面專家知識庫拿技能，不如讓 LLM 自己從玩的過程裡把技能抽取出來，再讓那套技能庫回頭改善決策**。

架構上，系統分成兩個 LLM 驅動的 agent，兩者構成一個共同演化的封閉循環。

**決策 Agent（Decision Agent, AD）**——這就是那個在遊戲裡做決策的傢伙。它維護一個當下的意圖標籤（intention tag）和一個活躍的技能計畫。每一個時間步，它從技能庫裡檢索候選技能、更新意圖、在需要的時候切換技能，最後執行一個原始動作。它靠兩個 LoRA adapter 分別訓練：一個負責**技能檢索**，另一個負責**動作生成**。

**技能庫 Agent（Skill Bank Agent, AS）**——這個 agent 不玩遊戲，它的任務是**把決策 agent 產生的軌跡資料轉化成可重用的技能**。它有一個四階段的技能pipeline：邊界提議（boundary proposal）、推論分割（infer segmentation）、合約學習（contract learning）、技能庫維護（skill bank maintenance）。它也配備三個 LoRA adapter，分別對應分割、合約、以及策展（curation）。

這兩個 agent 不是各做各的。**它們在同一個 GRPO（Group Relative Policy Optimization）優化迴圈裡共同更新**——更好的技能讓決策更強，更好的決策軌跡讓技能庫的品質提升，然後再循環回來。論文把這種設定叫做 co-evolution，譯作「共同演化」。

## 技能庫的四階段提煉流程

技能庫 Agent 怎麼把一段 raw 軌跡變成一個可以用、去用的技能？論文描述了一套相當完整的pipeline。

**第一關：邊界提議（Boundary Proposal）**。軌跡裡哪些地方可能是技能的邊界？演算法用啟發式偵測：述詞切換（predicate flips）、意圖標籤變化、獎勵訊號的尖峰、從執行原始動作切換到選技能的節點。這些都是「這裡可能做完一件事了」的跡象。這關卡的候選邊界數量會很大，所以設計上偏向「先大膽估、再精準篩」。

**第二關：推論分割（Infer Segmentation）**。有了候選邊界，模型要做選擇——哪些是真的技能邊界、哪些是噪音。這一步由一個專門訓練過的 segmentation adapter 把關。

**第三關：合約學習（Contract Learning）**。每一段分割出來的技能，要學的不只是「這技能做了什麼」，還要學「這個技能執行完之後，環境狀態會怎麼變」。論文把這個叫 effect contract——一種濃縮的效果預測器，描述技能執行後的狀態轉換。

**第四關：技能庫維護（Skill Bank Maintenance）**。技能庫不是只進不出的。新的候選技能進來之後，會經過 refinement（低品質技能改善）、merging（相似的技能合併）、splitting（過於寬泛的技能拆分）、以及 retirement（明顯過時或無用的技能下架）。這也是為什麼叫做「庫」而不是「列表」——它會自己淘汰。

## 實驗結果：8B 模型打贏了誰？

論文在六個遊戲環境裡測試 COSPLAY 的效能，包括單人遊戲 Benchmark 和多人社會推理遊戲。Base model 是 8B 規模的開源 LLM。

結果很有意思：COSPLAY 在單人遊戲上的平均 reward 提升了 **25.1%**，對比的 baseline 包括四個 Frontier LLM agent。這是一個不小的數字，考慮到 8B 模型在資源消耗上遠比這些 Frontier 模型便宜。

更值得注意的一點是：COSPLAY 的技能庫是從**無標籘的軌跡**裡自動長出來的，而不是靠專家標注。這降低了对大规模人工标注数据的依赖。

在多人社會推理遊戲上，COSPLAY 的表現同樣具有競爭力。這一點有點反直覺——技能庫 agent 的設計初衷是處理「單人長時間跨度」的任務，但結果顯示共同演化的機制對社會推理場景也有幫助。

## 框架的代價與真正的限制

COSPLAY 不是無中生有。它的技能庫需要**每個episode結束後重新跑一次pipeline**。這對即時線上遊戲場景不太友好，但對那種可以離線重播的任務環境（比如讓AI自己訓練自己）則完全可行。

另一個值得觀察的限制是：**技能的跨領域遷移能力還沒有被充分測試**。論文主要在遊戲環境做實驗，遊戲內的技能（比如「在某個狀態下執行躲避」）是否有辦法遷移到開放域的機器人操作或程式碼生成任務，目前並沒有答案。遊戲測試集是一個乾淨的起點，但不代表走出了實驗室。

另外，技能庫會「成長」這件事帶來了一個潛在問題：隨著技能庫越來越大，檢索延遲和技能選擇的準確度會成為新的瓶頸。這在論文裡幾乎沒有被討論，算是未來改進方向。

## 我的判斷

COSPLAY 最有意思的地方不是「25.1% 的提升」這個數字本身，而是它回答問題的方式：**不是靠更大的模型，而是靠讓模型學會重用自己的經驗**。

過去一年 agent 發展的主流方向是「推理時 compute 配置」——讓模型在回答之前先想久一點、想多幾步。但這條路有一個內在的極限：即使你想得再久，每一次想的起點都是乾淨的，沒有累積。COSPLAY 的做法是在推理計算之外，疊加一個「跨episode的結構化學習」，讓模型不只是推理者，也是一個會累積的學習者。

當然，co-evolution 的框架有一個還沒被回答的問題：這套共同演化機制，在真實世界的非遊戲任務（例如 tool-use agent 的 long-horizon 任務）上能不能 work？論文沒有測。但從架構邏輯上來看，只要有一個可以重複執行並收集軌跡的環境，共同演化的機制就能啟動。

如果 skill bank 的概念最後被證明可以跨任務遷移，那它會是 LLM agent 架構史上的一個重要模組——不只是一篇論文，而是一個可以被移植的設計思路。
