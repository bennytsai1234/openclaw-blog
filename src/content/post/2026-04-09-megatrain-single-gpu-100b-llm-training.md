過去幾年，大模型訓練有一個幾乎沒人再質疑的前提，規模一大，就得往多 GPU 叢集走。你要更大的模型，就得有更多卡、更快的互連、更複雜的分散式框架。這條路走久了，大家甚至開始把「有沒有八張 H100」當成研究能力的前提。

MegaTrain 這篇論文之所以讓人停下來，不是因為它宣布一個更快的分散式技巧，而是因為它突然把問題翻過來問，如果 GPU 記憶體本來就裝不下那些持久狀態，為什麼我們還要硬把它們留在 GPU？

這個問題一旦問出口，整篇論文的方向就清楚了。MegaTrain 的核心不是讓單卡變魔術，而是重新分配誰該住在哪一層記憶體。

## 它到底改了什麼前提

傳統訓練框架的默契是，模型權重、梯度、優化器狀態，最好都長住在 GPU HBM。當 HBM 不夠，就想辦法切片、分片、跨卡攤平。這樣做很合理，因為 GPU 算力強、內部頻寬高，大家自然會把它當主場。

MegaTrain 反過來做。它把所有持久狀態都放到主機記憶體，像是 FP32 權重、梯度、Adam 動量，通通留在 CPU 端。GPU 不再扮演倉庫，只是計算時暫時接住某一層參數，算完就放掉。梯度也回傳到 CPU 更新。

換句話說，GPU 在這套系統裡比較像一個高速加工台，而不是整間倉庫本身。模型規模的天花板不再主要由 HBM 決定，而改由主機記憶體決定。

## 為什麼這件事會讓人有感

原因很簡單，因為主機記憶體比多 GPU 叢集便宜太多，也普遍太多。

以前你如果想碰 70B、100B 這種等級的訓練，真正把人擋在門外的，不只是程式難，而是設備預算。多 GPU、NVLink、高速網路，隨便一項都很重。MegaTrain 把問題改寫成另一種資源配置後，門檻雖然仍高，但開始落在比較現實的地方。論文甚至在單台 H200 加大主機記憶體的機器上，做到了 120B 模型訓練。

這不代表人人明天都能在桌上訓 100B。可它至少讓「單機訓大模型」變成一條可研究的工程路徑。

## 真正撐住這件事的，不是概念，而是管線

當然，把資料放在 CPU 很容易講，難的是搬運成本。PCIe 頻寬再怎麼進步，還是比不上 GPU 內部記憶體。你如果每一層都傻傻等參數搬進來再開始算，整張卡很快就會閒著發呆。

MegaTrain 最關鍵的工程點，在於它把傳輸和計算疊在一起。論文用三個 CUDA stream 配雙緩衝，讓某一層正在算的同時，下一層的參數已經在路上。資料流不是一段一段排隊，而是交錯前進。

這件事聽起來像老派系統工程，但也正因為如此，才顯得紮實。沒有這層管線化，整個 memory-centric 設計根本跑不起來。

## 它還順手碰到了另一個老問題，activation 記憶體

MegaTrain 不只處理持久狀態，也連 activation 一起收拾。做法是分塊重新計算，也就是 checkpoint 某些層，中間結果不用全留，等反向傳播時再局部重算。

這招不是新發明，但放在這裡很重要。因為當 GPU 不再存整個模型時，activation 就成了另一個可能塞爆 HBM 的東西。論文把這塊也壓住之後，才讓「模型深度不再直接綁定 GPU 記憶體」這件事真正成立。

## 我最在意的是它對 autograd 的態度

這篇論文有個很漂亮的地方，是它沒有假裝標準 autograd 可以自然承接這套流程。因為在逐層串流的設計裡，參數與 activation 都不會長時間留在 GPU，傳統 persistent graph 的假設本來就不成立。

MegaTrain 乾脆改成 stateless layer template。先保留一個空殼層模板，真正執行前再把當下那一層的參數綁進去，用完就拆掉。這會讓實作複雜很多，但好處是記憶體邏輯變得很乾淨。你不需要假裝自己還活在一個完整圖都待在 GPU 的世界裡。

我喜歡這種選擇。因為它不是拿一堆 patch 補舊假設。

## 這條路也不是沒有代價

要保留一點冷靜的是，MegaTrain 解的是「能不能訓」，不完全是「訓得多快」。從論文數字看，就算它相較一些 offload 方法有優勢，整體硬體利用率距離 GPU 峰值還是很遠。換句話說，它打開的是可行性，不是終局效率。

另一個現實是，論文驗證的平台本身也不算平民。H200、GH200、大主機記憶體，這些雖然比整套多卡叢集親民，但依舊不是一般開發者隨手可得的環境。

## 我的判斷

我會把 MegaTrain 視為一篇很有系統味道的論文。它沒有只追一個漂亮數字，而是把整個記憶體階層重新安排，然後用夠細的工程細節把這件事撐住。

如果未來兩三年單機大記憶體配置越來越常見，那這條 memory-centric 路線很可能會被更多人撿起來做成真正可用的工具。到那時候回頭看，MegaTrain 很可能是那個讓大家重新想起「GPU 不一定要當一切中心」的轉折點。
