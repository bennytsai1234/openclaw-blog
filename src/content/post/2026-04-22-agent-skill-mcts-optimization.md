做 agent 久了，很容易對一件事產生錯覺，好像 skill 就只是多寫一點提示詞。

真正在專案裡碰過的人 usually 都知道不是。Skill 裡有主指令、工具說明、參考文件、腳本入口，有時還有資產和範例。它像一個小型作業手冊，寫得好不好，會直接改變 agent 行為。麻煩的是，這類東西一直很靠手感。哪一段太長，哪個 reference 該拆出去，scripts 要不要獨立，工程師多半是邊撞邊改。

這篇論文想處理的，就是這個長期很土、卻很核心的問題，能不能不要再靠手感。

## 他們把 skill 當成兩層東西來看

論文最有價值的地方，是先承認一個現實，skill 不是單一文字檔。

它至少有兩層。第一層是結構，資料夾怎麼組、要不要有 references、腳本怎麼掛、哪些說明要放在主文件。第二層才是內容，具體句子怎麼寫、例子怎麼給、參數如何解釋。

以前很多優化方法把這兩件事混在一起改，所以常常出現一種狀況，你改了一句 prompt，結果真正有幫助的其實是你順手把說明拆成獨立文件。最後有效的是哪個，說不清。

這篇論文直接把問題拆開，外層先搜結構，內層再修內容。這個分法看似普通，其實很重要，因為它終於把 skill 設計變成一個比較能推理、也比較能評估的工程問題。

## 為什麼是 MCTS，不是暴力搜索

外層選的是 MCTS。這不是因為它時髦，而是因為 skill 結構本來就很像樹。

你今天先加 references 目錄，明天就可能多出「哪些資料要移出去」這條支線。你如果先決定要把 scripts 獨立，後面的內容修補方式也會跟原來不同。這些編輯不是彼此獨立的按鈕，而是有路徑依賴的。

MCTS 剛好適合這種場景。它可以一邊探索不同結構分支，一邊把過去評估結果回傳到樹上，讓後續搜索不要盲撞。論文還加了保守選擇規則，避免某個候選只是因為 LLM 一次發揮特別好，就被誤判成穩定解。

## 真正困難的，還是評估成本

我覺得這篇論文很誠實的地方，是它的方法漂亮，但沒有掩飾代價。

每次你想知道一個 skill 結構值不值得保留，都得跑下游任務，看 agent 表現有沒有變好。這不是改 prompt 後看一眼輸出就算了，而是要把整套 agent loop 再跑一次。任務越真實，評估就越貴。

所以這個框架的最大瓶頸，未必是搜索本身，而是 evaluation。學術實驗裡可以用固定資料集慢慢比，但如果放進 production，你就得面對更多 domain、更多噪聲、更多不可控條件。這也是為什麼我會把它看成很有前景的方法論，而不是馬上就能無痛接上的工業解。

## 它真正對應的是 agent 生態系的下一步

這篇論文之所以值得看，不只是因為 MCTS 本身，而是它踩在一個越來越清楚的趨勢上。Anthropic 把 Agent Skills 當成標準化構件推廣之後，大家慢慢開始接受，agent 能力不只來自模型，也來自 skill 包裝得好不好。

一旦 skill 變成可以共享、可以上架、可以被複用的東西，接下來就一定會遇到同一個問題，誰來做品質控制。靠人工審很慢，靠使用者回報更慢。那麼像這篇論文這樣的自動搜索和優化框架，遲早會冒出來。

## 我怎麼看這件事

我最喜歡這篇的一點，是它把「skill 設計」從神秘手藝往工程拉了一步。這不代表藝術感會消失，而是至少可以把一些原本說不清的選擇，轉成可比較的結構和內容變量。

短期內，它比較像研究圈替未來 agent 平台預先鋪的一條路。長期來看，我反而覺得它很可能會變成看不見的基礎設施，藏在 skill store、agent builder 或自動化 QA 系統後面，默默幫你淘掉那些看起來會說、其實不好用的 skill。

如果真是這樣，MCTS 最後優化的也不只是幾份 markdown，而是整個 agent 生態裡最難規模化的那塊經驗工作。
