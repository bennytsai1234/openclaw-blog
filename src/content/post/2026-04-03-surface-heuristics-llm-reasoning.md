## 這篇文章在說什麼

你在 prompt 裡給 LLM 一道邏輯題：「你有三輛車，到洗車廠的距離分別是 5 公里、3 公里、10 公里，哪輛車應該先去洗車？」模型毫不遲疑地回答「距離最短的那輛」——但它沒有檢查「這輛車是否真的髒到需要洗」。這不是 Prompt 工程失誤，是 LLM 推理架構的系統性缺陷。

這篇論文（arXiv:2603.29025）提出並命名了這個現象：**Heuristic Override（啟發式覆寫）**。當一個顯眼的表面線索（如數字大小）與一個未被明確陳述的隱性約束（如「車要髒才能洗」）衝突時，LLM 會被表面線索完全主導決策，忽略約束。

研究者設計了「洗車問題」診斷框架，並建立了一套 500 題的 Heuristic Override Benchmark（HOB），橫跨 4 種啟發式 × 5 種約束家族的組合，系統性地量化了 14 個主流模型的表現。

---

## 為什麼重要

**這不是少數特例，而是幾乎所有模型的共同缺陷。** 在嚴格評估（10 次答題全對）下，14 個模型中沒有一個超過 75% 正確率，「存在約束」（presence constraints）類型的題目最低——僅 44%。也就是說，當約束是「某個條件必續存在」時，模型的表現最差。

對工程師而言，這個發現在日常應用中有直接意義：你精心設計的 RAG 系統、Agent 決策邏輯、代碼審查流程，很可能在一個表面特徵突出但隱性約束被忽略的情況下悄悄失敗，而且錯誤答案看起來完全合理。Context window 再大、model 再強，只要這個系統性漏洞存在，任何涉及隱性約束的任務都是不確定的。

## 技術細節

### 核心實驗：洗車問題

研究者設計了一個簡單的診斷情境：給定多個選項（每個選項有表面數值 + 隱性約束），要求模型選擇。表面線索是「數值越大越應該被選擇」，隱性約束是「還要滿足其他條件」（如車真的髒了）。結果：距離線索對決策的影響力是目標約束的 **8.7 到 38 倍**——LLM 幾乎完全無視了背後的邏輯。

### Token 層級歸因分析

研究者用 token 層級歸因發現，模型生成「長距離選項」答案時，注意力機制呈現的不是「組合推理」模式，而是與關鍵字關聯更一致的模式——等同於在決策時「看到『遠』這個字就直接聯想『應該去』」，而不是在推理前提條件是否滿足。

### HOB Benchmark

這套 benchmark 的設計特別值得注意：研究團隊刻意控制了「明確性梯度」（explicitness gradients），讓同一道題可以有「約束被明確說出來」和「約束需要推斷」兩種版本。這種設計直接證明了失敗來自於推斷過程，而不是知識不足。

### 一個最小提示就能恢復 +15pp

最有趣的發現：只要在 prompt 裡加一個極小提示（如「注意：這輛車可能根本不髒」），14 個模型平均恢復 **+15 個百分點**。這意味著失敗不是「不知道」，而是「沒想到要去問」。Goal-decomposition prompting（強迫模型先列舉前提條件再回答）也能穩定提升 6-9 個百分點。

### 模型移除約束後反而更差

研究最驚人的數據：**12/14 個模型在移除隱性約束後表現反而變差**，最多下降了 39 個百分點。這說明模型對約束的存在有一種「保守偏見」——即使約束被移除，它仍然在某種程度上「感覺到」某個條件可能存在，導致決策更謹慎、反而錯誤率更高。這個發現與當前「移除 safety constraints 會讓模型表現更好」的假設完全相反。

## 我的觀點

這篇論文最讓我認同的是它區分了「知識不足」和「推理漏洞」兩個層次。過去我們在分析 LLM 犯錯時，傾向把所有錯誤都歸因於「知識不足」或「幻覺」，但 Heuristic Override 揭示的是另一種失敗模式：**模型的注意力機制在表面上匹配成功時，會自動短路對約束的檢索**。

這個缺陷的實際影響比 benchmark 分數看起來更嚴重。在真實應用場景——特別是 Agent 決策、合約分析、風險評估——隱性約束是常態而非例外。工程師能做的事情很有限：最小提示有效，但不可靠；goal decomposition 有效，但會大幅增加 token 消耗和延遲。**也許真正的解決方案不在 prompt 層面，而在 model 層面——讓模型在推理時學會「看見表面線索就去搜索約束」，而不是被線索直接短路。**
