你有沒有想過一個問題：語言模型聲稱是「壓縮機器」，把龐大的訓練資料壓進參數裡，但一個 token 的 KV 快取卻比它自己還要大。

Llama-2-7B 的例子很有說服力。一個 token 的整數表示不過 2–4 個位元組，但對應的 KV Cache 佔用量約 512 KB——膨脹超過十萬倍。緩存一本《愛麗絲夢遊仙境》的上下文，大約要消耗相當於整個模型權重本身 1.4 倍的記憶體。這聽起來像是某種系統設計失誤，而不是尖端 AI 系統該有的樣子。

問題的核心在於：Transformer 的自迴歸生成需要反覆存取所有先前 token 的 Key 與 Value 狀態。現有方法主要著眼於「時間軸」——哪些舊 token 可以刪掉、哪些可以壓縮。但 Apple 的研究團隊指出，還有一個維度幾乎沒人认真處理：**深度維度**。也就是：每一層是否真的需要自己專屬的 KV Cache？

## 舊方法碰到的牆

在這篇論文之前，已經有人嘗試跨層共享 KV Cache。問題在於代價。

一種做法是訓練一個單獨的 Encoder，統一生成所有層共享的 KV 表示。但每次更新 Cache 都要跑一次完整的前向傳遞，延遲直接爆表。另一種做法是直接用淺層的輸出估計深層狀態，結果 prefill 階段需要多次前向傳遞才能達到合理的準確度，時間成本同樣無法忽視。

論文裡稱這些方法「suffer from reduced throughput or increased time-to-first-token」，說得很坦白。也就是說，跨層共享這條路不是走不通，而是之前走的代價太大，大到沒有人真的想在生產環境裡用。

那問題就變成：有沒有辦法讓模型自己適應跨層共享，而不是靠外部技巧硬湊？

## R-CLA：讓模型在訓練時就習慣借用

Random Cross-Layer Attention，簡稱 R-CLA。概念很乾淨——在訓練時，每一個層有一定的機率不去用自己的 KV 狀態，而是借用前面某一層的。

具體來說：對於第 $l$ 層，在每一次前向傳播時，以機率 $p$ 執行標準自注意力（用自己的 KV），以機率 $1-p$ 強制去讀取隨機選擇的前面某一層 $l' < l$ 的 KV 狀態。這個隨機過程模擬了一種「結構性快取缺失」——不是因為硬體限制被迫放棄，而是訓練本身就一直在經歷這種「借來的」 attention。

論文把這個過程類比成一種隨機的正則化。模型在訓練時不知道這次到底是用自己的 Cache 還是借來的，因此必須學會從早期層的一般性語意表示中萃取出足够維持預測質量的資訊。換句話說，它被迫去理解「到底需要什麼」而不是「從哪一層拿」。

這種訓練方式帶來一個重要的特性：**彈性**。模型在部署時可以選擇快取每隔第 4 層、放棄 50% 或 75% 的層級，因為它早就被訓練成應付這種狀況。不同硬體環境可以用同一個模型，不需要為記憶體受限的邊緣裝置和高效能的資料中心分別訓練不同的版本。

## 數字說了什麼

研究團隊在多個模型家族上做了實驗，包括 Llama-2-7B、Llama-3-8B 和 Qwen3-8B。圖表顯示，用 R-CLA 訓練過的模型，在僅保留 25% 層級快取的情況下，Question-Answer 任務的 F1 分數與完整快取的基準線相差無幾。對照組——沒有經過 R-CLA 訓練、直接共享快取的模型——在同樣的保留率下，分數下滑幅度明顯更大。

更有意思的一點是：較大的模型在完整快取時，R-CLA 版本居然經常**表現持平或略好**於標準模型。團隊推測這可能是一種正則化效應——隨機借用早期層表示的做法迫使模型不要過度依賴特定層級的特徵，反而讓泛化能力更健康。對資料受限的場景來說，這是一個意外的紅利。

記憶體節省也相當可觀。Llama-2-7B 原本每 token 需要 512 KB 的 KV Cache，採用 R-CLA 之後，快取層級減半，記憶體佔用直接砍半；對於 Llama-3-8B 這類已透過 Grouped-Query Attention 優化過的模型，進一步的跨層共享同樣有效。

## 這個方向被低估了嗎

我的看法是，這篇論文的核心價值不在於「能省多少記憶體」這個數字，而在於它重新定義了問題的框架。

之前多數 KV Cache 的研究集中在時間軸：哪些 token 該留、哪些該刪。這些方法確實有效，但它們有一個根本的限制——token 的重要性是 query-dependent 的，這一秒不重要不代表下一秒不需要。這就是為什麼像 H2O、SnapKV 這些方法都必須在推論時動態重新計算，而且永遠有重新遺失重要資訊的風險。

深度維度的共享則沒有這個問題。每個層的 KV 狀態都是完整的語意表示，不存在「這個 token 在那個語境下是否重要」這類動態判斷。共享第 $l$ 層的快取給第 $l+2$ 層用，並不需要知道任何未來的 query——它只是一個確定的結構性選擇。這讓部署時的策略選擇可以完全繞過 query-dependent 的計算 overhead。

不過，這不代表它沒有代價。論文的 Limitations 章節提到，R-CLA 適配需要對模型做額外的訓練或微調，這在某些情境下（特別是當你只有一個已經訓練好的模型、沒有資料或計算預算再做額外訓練時）會是障礙。這也是為什麼論文特別強調 R-CLA 在 pre-training 或 fine-tuning 階段加入是最順暢的——而不是事後對一個已經部署的模型做修補。

## 結語

回到最初的問題：語言模型真的是「壓縮機器」嗎？從參數的角度來說，是的。但從 KV Cache 的角度來說，它根本不像。反諷的是，這個看似矛盾的現象，其實是 transformer 架構的基本限制造成的——每一層都要維護自己的一套狀態表示，卻又高度重疊。

R-CLA 嘗試處理的正是這種結構性的浪費。它不是試圖發明更聰明的 eviction 策略，而是在更根本的層次上重新訓練模型，讓「共享」不再是損失、而是可預期的能力。這條路目前還在 research stage，距離實際進入某個模型的 serving pipeline可能還有一段距離。但我認為它揭示的方向值得關注——當我們把所有力氣都花在時間軸上的快取優化時，深度維度的冗餘可能才是那個被長期忽略的更大改進空間。
