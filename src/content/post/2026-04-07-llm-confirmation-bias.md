很多人第一次用 LLM 做推理任務時，都會有一種錯覺，模型好像很理性。

它講話完整，步驟分明，甚至會自己列出假設、推導、結論。可只要你真的把它放進多步互動任務，讓它一邊猜、一邊測、一邊修正，你很快就會發現另一件事，模型未必在找真相，它常常只是在找能證明自己沒錯的東西。

這就是 NYU 這篇研究最讓人不安，也最值得看的地方。

## 經典心理學問題，拿來照 LLM 很剛好

研究團隊用的是 Wason 的 2-4-6 任務。規則很簡單，你看到一組數字，必須透過提出新例子來猜背後的規則。環境只會回你「符合」或「不符合」。

這個任務之所以經典，是因為人類很容易犯一個錯。我們想到一個可能規則後，就會一直拿能支持這個規則的例子去試。這樣看起來很積極，其實很沒效率。因為你真正需要的，不是更多支持，而是反例。你要想辦法讓自己的假設倒下，才能比較快接近正確答案。

研究者把這套任務交給 11 個不同的 LLM，結果很一致，模型也會犯同樣的毛病。它們會偏好提出支持當前假設的測試，而不是拿可能推翻自己的例子去試。

## 這件事為什麼不只是心理學趣聞

如果 LLM 只是被拿來回答單題知識問答，確認偏誤可能只是小瑕疵。可一旦它被拿來當 agent，問題就完全不一樣了。

Agent 要做的事情，往往不是單步作答，而是持續形成假設、呼叫工具、讀取結果、修正方向。這整條鏈最怕的，就是前面想錯，後面還一直替自己找證據。那不是普通的 hallucination，而是一種會自我加固的錯誤。

你可以把它想成一個很勤奮、也很自信的實習生。他不是不做事，他會一直做。但他做的每一步，都在幫同一個錯誤方向添柴火。這比單純答錯更麻煩，因為流程看起來是動的，甚至很有條理。

研究裡一個很重要的發現是，確認偏誤越明顯的模型，任務成功率通常越低。這說明模型規模變大，不會自動把這種傾向洗掉。你可以有更大的參數量，但依然不擅長懷疑自己。

## 有意思的不是發現偏誤，而是修正方法很老派

我最喜歡這篇論文的一點，是它沒有急著發明一個新的機器學習魔法名詞。研究者反而回頭借用了人類心理學裡已經存在的方法。

第一種叫 Think-in-Opposites，簡單說就是刻意往相反方向想。你不要只問「什麼例子符合我的假設」，而是要問「什麼例子最可能打臉我」。

第二種叫 Dual-Goal。模型不能只猜一個規則，還要同時考慮它的補集。這會逼它維持兩條競爭中的解釋，而不是太早押單邊。

結果很實際，任務成功率從 42% 提到 56%。數字不算誇張，但已經足夠說明方向是對的。更重要的是，後續透過知識蒸餾，這種「反確認偏誤」的行為模式還能被留進模型，不必每次都靠 prompt 提醒。

## 這對 agent 設計有什麼提醒

現在很多 agent 系統在追求的是更長的上下文、更多工具、更快的流程編排。但這篇研究像是在說，如果模型沒有內建一點「自我懷疑」的能力，你只是讓它更有效率地走錯路。

這對產品設計很有啟發。也許我們該把一些反證機制，直接做成預設流程。像是在重要步驟後要求模型列出一個可能反例，或者在採取行動前強制它提出第二種解釋。這種設計看起來增加了一步，但在高風險任務裡，可能比一味加速更有價值。

對評估也是一樣。過去我們常用最終正確率衡量模型，現在也許該多看一個維度，它在探索過程中有多願意推翻自己。因為真實世界的問題，通常不是一次猜中，而是在連續試探裡慢慢接近。

## 我自己的保留

當然，這篇論文還是有它的邊界。Wason 任務很乾淨，回饋也明確。現實裡很多問題不會直接告訴你「符合」或「不符合」，而是給一堆模糊訊號。模型能不能把這種反證習慣帶到真實軟體開發、醫療判斷、研究助理這些場景，還需要更多證據。

但我不覺得這削弱了它的重要性。反而因為任務夠乾淨，我們才第一次比較清楚地看見一件事，LLM 的推理缺陷，未必只是算力不夠或資料不夠，也可能是它在探索策略上就和人一樣，會偷懶，會過早相信自己。

## 如果要我用一句話總結

這篇研究真正讓人記住的，不是「AI 也有確認偏誤」這個標題，而是另一層更麻煩的事實，很多我們以為屬於人類認知的老問題，到了 LLM 身上，並沒有消失，只是換了一種更像推理的語氣出現。

而這件事一旦成立，我們之後設計 agent、評估模型、做安全防護的方式，都得跟著變。因為問題已經不是模型會不會想，而是它會不會在想的時候，主動懷疑自己。
