假設你今天遇到一個很簡單的 bug：某個函式裡 `range(len(x) - 1)` 應該是 `range(len(x))`，就這一個 off-by-one 錯誤。你把程式丟給 AI 幫你修，它處理完之後拿回來看——功能確實是對的，測試也全過了。但整個函式被整個重寫了：多了 `None` 檢查、`np.asarray` 轉換、finite-value mask、新增輔助函式、變數名稱被改過、連繪圖邏輯都換了一套。

這種「改得比需要的更多」的現象，作者稱之為 **Over-Editing**。

這個問題值得在意的原因很直觀。在軟體工程的世界裡，多數時候我們不是在寫新程式，而是在既有程式上施工。既有程式已經被團隊理解過、review 過、確定是那個樣子的。當你需要修一個小 bug，別人的期待是你只動那一行，其他東西全部保留——因為多改一行，就多一個讓 review 委員重新理解整個函式的理由，也多一個在部署後產生預期外行為的機會。

然而現在這些 AI 工具的做法幾乎完全相反。它們像是那種「好意但控制狂」的同事：你要一杯水，它給你順手把整個廚房重新整理了一遍，杯子洗了但整個佈局都變了。

## 一個看起來很直覺的問題，背後其實沒那麼簡單

這個研究的起點是一個看似簡單的問題：LLM 在修改程式碼時，會不會過度改寫？

但要回答這個問題，得先定義「什麼叫過度」。研究團隊從 BigCodeBench 拿了 400 個問題，然後用程式化的方式對每個正確解答施加「腐蝕」——翻轉比較運算子（`<` 變成 `<=`）、把加號換成減號、把布林值從 `True` 改成 `False`，讓解答變成錯誤版本。被腐蝕的程式碼仍然語法正確，只是跑不過測試。這樣一來，「正確解答」就是腐蝕的完全反轉，也就是最小變動。

這個設計聰明的地方在於：它同時解決了「ground truth 是什麼」和「什麼算 minimal edit」這兩個問題。最小編輯就是那個精確反轉，其他任何多餘的變動都是 over-editing。

研究團隊用兩個指標來測量 over-editing 的程度：

**Token-level Levenshtein Distance**：把修改後的程式碼和被腐蝕的版本都先做 tokenize，再用 Levenshtein 距離比對。不做 tokenize 的話，把變數名稱 `add` 改成 `someotherfunctionname` 會被計算成 19 個字元的距離，但 tokenize 後兩者各是一個 token，距離只有 1。

**Added Cognitive Complexity**：衡量修改後程式碼多了多少認知複雜度。認知複雜度會計入巢狀深度、迴圈、條件判斷等讓人類讀者需要「在腦中多持有一個狀態」的元素。因為所有腐蝕都只改數值、不改結構，所以一個忠實的修改應該讓認知複雜度完全不變。任何增加的部分都是模型擅自加進來的。

## 測量結果：所有模型都在 Over-Edit，程度差異很大

先看六個主要模型在推理模式下的表現：

| 模型 | Pass@1 | Normalized Levenshtein | Added Cognitive Complexity |
|------|--------|------------------------|---------------------------|
| GPT-5.4 (High) | 0.723 | **0.395** | **2.313** |
| Claude Opus 4.6 | **0.912** | **0.060** | **0.200** |
| Gemini 3.1 Pro Preview | 0.858 | 0.145 | 0.501 |
| GLM 5 High | 0.859 | 0.099 | 0.320 |
| DeepSeek R1 | 0.820 | 0.232 | 0.673 |
| GPT-5 High | 0.713 | 0.438 | 3.832 |

數字愈低代表愈少 over-editing。GPT-5.4 的 Levenshtein 是 0.395，意謂它的輸出與「最小改動」之間的 token 距離是 0.395 倍於整個程式的長度——這已經不是「多改一點」的範圍了。GPT-5 High 更是來到 0.438，Added Cognitive Complexity 高達 3.832，等於把一段本來相對好讀的程式改成了需要讀者一直持有多個狀態的版本。

而 Claude Opus 4.6 是少數同時兼顧正確性與編輯精準度的模型：Pass@1 最高（0.912），Levenshtein 只有 0.060，Added Cognitive Complexity 只有 0.200。三個指標全部領先。

## 推理模型比非推理模型更容易 Over-Edit

研究有一個發現很值得拿出來討論：預設行為下，**推理模型（reasoning model）比非推理模型更容易 over-edit**。

把同一家模型公司的推理與非推理版本拿來比對，多數情況下推理版本的 Levenshtein 數值都更高——意味著它改得更多。GPT-5.4、DeepSeek V3、Gemini 3.1 Pro Preview、Qwen 3.6 Plus、Kimi 2.5 全部都是推理版本改得更多。只有 Claude Opus 4.6 是例外，推理版本反而比非推理版本收斂得更精準。

為什麼？研究者的推測是：推理能力讓模型有更多「餘裕」去優化它認為「可以更好的地方」。一個簡單的 bug fix，在模型眼裡可能是「順便把變數命名現代化、加入輸入驗證、把函式重構一下」的機會。推理鏈越長，這種「優化」衝動累積得越多，最後變成一個幾乎把整個函式重寫的輸出。

這帶出一個很大的問題：AI  coding 工具之所以被信任，很大原因是「它們只做你要求的事」。但現在的推理模型預設行為是「順便把其他的也改好」，等於把這個信任前提悄悄破壞了。

## 直接叫模型「不要亂改」，多數時候有效

有趣的是，只要在 prompt 裡加一句：「請尽量保留原始代码及其逻辑」，幾乎所有模型都會收斂 Levenshtein 距離，而且推理模型的改善幅度比非推理模型更顯著。

這件事有兩層意思。第一層很直觀：明確的約束是有效的。模型不是不知道你在乎「只改需要的」，而是預設行為沒有把这件事當成目標，加了 prompt 就糾正回來了。

第二層更重要：同一套推理能力，在有約束的時候反而變成優勢。推理模型因为更强的 instruction following，所以約束丢下去的時候，它服從得也更徹底。圖表數據顯示，推理模型在 explicit prompt 模式下幾乎全部低於非推理模型的 Levenshtein——代表一旦被告知目標，推理模型反而是最聽話的。

也就是說，over-editing 不是推理模型的根本限制，只是預設行為。可以被約束扭轉，只是預設沒有這個約束。

## RL 訓練能把模型教成「精準編輯器」，而且不影響其他能力

研究的另一個重點是：能不能透過訓練，讓模型天生就更傾向最小編輯，而不是每次都要靠 prompt 約束？

團隊以 Qwen3 4B 2507 Instruct 為基底，試了四種訓練方式：

**SFT**（監督式微調）：直接拿程式化腐蝕的資料集做 supervised fine-tuning，一開始看起來效果驚人——Levenshtein 降到 0.002，認知複雜度增加為 0。但這只是模型學會了「反轉腐蝕」這件事本身，而不是學會了「最小編輯」的原則。一旦測試集裡的腐蝕類型與訓練集不同，Pass@1 立刻崩跌到 0.458——模型根本不會修 bug 了，只是機械地把「看起来像腐蝕」的模式反轉回去，無論那個反轉能不能讓測試通過。

**rSFT**（rejection-sampled SFT）：模型自己生成多個版本，只訓練其中 Levenshtein 最低的那幾個。這種做法比純 SFT 更穏健，但仍有輕微的災難性遺忘。

**DPO**（直接偏好優化）：在模型自己生成的多个版本中，用偏好學習拉高 Levenshtein 最低的、壓低最高的。效果比 rSFT 再好一點點，泛化也比較乾淨。

**RL**（強化學習）：結合功能正確性獎勵與 Levenshtein 獎勵的複合 reward。這是唯一在泛化實驗中全面勝出、且完全沒有災難性遺忘的方法。訓練過的模型在 LiveCodeBench 上與原始模型表現幾乎相同，但在最小編輯指標上領先所有 baseline。

這個結果對整個 LLM coding agent 領域是個好消息。它說明「精準編輯」這件事可以被專門訓練出來，而且代價不會是犧牲其他 coding 能力。但它也暗示了一個現實：多數模型預設的 over-editing 行為，其實是 RL 訓練過程中沒有把「編輯精準度」當成目標而產生的副作用。說穿了，是目標函式裡沒寫進這條。

## LoRA 就能解決，不需要全參數微調

一個對實務有意義的觀察：即使只用 rank 64 的 LoRA，Levenshtein 就能降到與 full RL 相近的水準（0.065 vs 0.050）。rank 64 的 LoRA 在 LiveCodeBench 上的退化也幾乎是零（Delta +0.001），代表它幾乎不影響其他 coding 能力。

這件事的實際意義是：如果你想在既有模型上讓它更精準地做 code editing，不需要重新訓練一個新模型或負擔全參數微調的成本。LoRA rank 64 已經足夠讓那個「控制狂同事」變成一個懂得節制的幫手。

當然代價是：這項研究目前只在 BigCodeBench 的程式化腐蝕環境下驗證。真實世界的 code editing task 複雜得多，context 結構差異很大，一個在人工構造環境中學會「最小編輯」的模型能不能直接轉移到 production codebase 還是未知數。但方向是明確的。

## 結語

Over-editing 這個問題，表面上只是「模型順手改太多」，但往深看它其實影響了 AI coding 工具在 brown-field 軟體開發中的可用性。當一次簡單的 bug fix 產生 30 行的 diff，而每一行都是 review 時需要重新確認安全性的陌生邏輯，這種工具產產产生的摩擦反而會侵蝕團隊對它的信任。

研究告訴我們幾件事：GPT-5.4 愛改最大、Claude Opus 4.6 最節制；推理模型預設更會 over-edit 但也更能被約束；RL 訓練可以把「精準編輯」教進模型而不影響其他能力；而且這件事不需要大模型，小規模 LoRA 也許就足夠了。

但或許最值得記住的是這句話：模型的預設行為反映的是訓練目標，而不是能力上限。當你在 prompt 裡說「不要改多的」，它就能做到——這件事本身，就說明了多數模型現在的 over-editing，其實是目標函式沒寫對的問題，而不是模型做不到。
