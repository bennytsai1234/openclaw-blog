有些版本更新，你看完 release note 會覺得聲勢很大，真正裝上去卻沒什麼感覺。也有些更新剛好相反，名字看起來平淡，真正用幾天才發現，手感完全不一樣了。Gemini CLI 0.37.0 比較像後者。

這次它沒有丟出一個足以改寫市場格局的新能力，沒有突然多一個神奇模式，也不是模型等級的升級。它做的是更樸素的事，把幾個每天都會撞到的摩擦點慢慢磨平。對長時間泡在終端機的人來說，這種更新通常比一張漂亮的功能海報還值錢。

## 為什麼我會把這版看得比表面上重要

Gemini CLI 這類工具，競爭點從來不只是模型本身。大家都有 code assist、都會跑工具、都能接 MCP。真正拉開差距的，往往是那些反覆發生的小麻煩有沒有被處理掉。

0.37.0 裡最值得注意的，就是它開始處理這些麻煩。像是同一個專案裡重複授權、終端畫面閃爍、專案記憶資料放在不對的地方。

## 這次最有感的更新，是審批終於記得上下文

以前用這類 agent 工具，最容易出現一種荒謬感。你明明在同一個專案目錄、做的是同一類操作，卻還得一再重複回答「你確定要這樣做嗎」。第一次會覺得安全，第三次開始煩，第十次只剩機械式點頭。

Gemini CLI 0.37.0 加入的 context-aware persistent policy approvals，解的就是這個問題。它會根據專案目錄記住你先前批准過的政策。也就是說，信任不是整台機器共用，而是跟工作上下文綁在一起。

這個設計很像一個成熟工具該有的樣子。它不是一味放寬，也不是一味保守，而是承認熟悉專案和陌生資料夾，本來就不該被同等看待。

## TerminalBuffer 看起來小，實際上很傷體感

另一個我覺得重要的更新，是 TerminalBuffer 模式。

如果你平常常讓 CLI agent 跑長輸出，應該對那種畫面重繪閃爍很熟。它不是不能用，但就是一直提醒你這東西還很粗。尤其在滾動瀏覽大量內容、看長 diff、追 shell log 的時候，閃爍本身就會打斷注意力。

0.37.0 把這個問題拉出來處理，代表團隊開始認真看待 UI 不是附屬品。終端機當然不是圖形介面，但不代表體驗可以隨便。對很多人來說，CLI 是一整天都打開著的工作表面。畫面穩不穩，真的有差。

## 專案記憶移到 tmp，這是很工程的修補

有些改動不會出現在宣傳文案最前面，卻很值得點頭。像這次把專案記憶目錄改放到 tmp，就屬於這類。

過去如果資料直接留在專案根目錄，最尷尬的事不是看起來亂，而是容易被不小心加進版本控制。AI 工具的中介資料常常混著上下文、摘要、操作紀錄，對私有專案尤其敏感。把它們從工作樹挪開，雖然只是路徑調整，背後其實是在補一個很常見的人類失誤面。

這也是我對成熟度的其中一個判斷。不是功能有多炫，而是它有沒有開始預設使用者會犯錯，並提前替你把坑填掉。

## Gemini CLI 眼下的定位，其實越來越清楚

如果拿它跟 Claude Code、OpenAI 這些同類工具一起看，Gemini CLI 的優勢其實一直都不是一句「它很強」可以講完。它的核心吸引力有三個。

第一，context window 大。當你真的在處理大專案、長對話或需要吃很多背景資料的任務時，這件事不是規格表裝飾，而是決定你要不要一直切上下文。

第二，MCP 支援自然，而且開源。這代表你不只是在用一個工具，也是站在一個能被外部擴充的生態上。對喜歡自己接管線的人來說，這點很重要。

第三，headless 與 CLI 工作流本身做得不差。`gemini -p` 這種模式很容易被接進腳本、CI、批次工作。

## 但它現在還不是一個可以完全放心的穩定選項

也要老實說，0.37.0 仍然是 preview。這代表它的定位更像是給願意先吃新功能的人，而不是要每個人立刻搬過去。你如果是在重視穩定性的生產流程裡，0.36.0 可能還是更保守的選擇。

另外，Gemini CLI 雖然進步很快，但也還在一種產品形狀持續變動的階段。CLI 參數、設定名稱、記憶行為，這些都有可能在數個版本內繼續調整。

## 我的感覺是，它終於開始在意每天的那八小時

我會把 Gemini CLI 0.37.0 看成一個訊號。不是因為它做了驚天動地的更新，而是因為它開始把注意力放在真正的工作時間裡。不是那五分鐘的初見體驗，而是你把它開著一整天之後，哪些地方會讓你想關掉。

上下文感知審批、TerminalBuffer、記憶目錄搬家，這些都不是 headline 級功能。但也正因為如此，它們更說明團隊已經開始面對一個更難的問題，怎麼把代理工具磨成真的能陪你工作的工具。

如果你最近剛好在觀察 Gemini CLI，0.37.0 是一個值得安裝來感受一下的版本。很多時候，工具是不是成熟，不是看它第一次能做什麼，而是看它第二十次還會不會惹你生氣。
