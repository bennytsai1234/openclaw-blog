很多工程師對模型品質下降的直覺，其實很準。

你不一定立刻說得出是哪裡怪，但你會開始感覺它變急了。該先讀檔的地方沒讀，該多看兩步的地方直接動手，明明同樣的提示詞，以前能穩穩做完，現在卻常常中途卡住，甚至往反方向走。

今年初關於 Claude Code 的爭議，大概就是這種集體直覺慢慢累積出來的。最終把問題講清楚的，不是官方說明，而是一位重度使用者整理出的數據。

## 這不是單純的情緒抱怨

這份分析之所以值得看，是因為它不是幾張截圖、幾段主觀感想，而是從幾千個 session 裡挖出趨勢。涵蓋的資料量夠大，時間也夠長，讓原本像都市傳說的東西開始有了輪廓。

爭議的核心很簡單，2026 年 2 月到 3 月之間，Claude Code 似乎不只是把 thinking 隱藏起來，而是連真正可用的 thinking 深度都一起縮了。

這兩件事差很多。

把 thinking 內容藏起來，理論上只是介面決策。可如果 thinking budget 也跟著被壓縮，受影響的就不是觀感，而是模型實際完成複雜工程任務的能力。

## 下降不是抽象的，它會長成幾種具體行為

最明顯的一個信號，是模型越來越常在沒讀完檔案前就編輯。這種行為在簡單改字串時也許還能混過去，但只要牽涉多檔案、共享型別、隱含約定，它幾乎一定會撞牆。

再來是整檔重寫的比例上升。以前比較像手術，改局部，保留上下文。後來越來越像拿大刷子直接重塗。從某種角度看，這是模型在節省思考成本。它不想精準定位，就乾脆整片改掉。問題是，這種方式在真實工程裡很危險，因為你很容易連周圍本來沒問題的東西一起弄亂。

還有一個很有畫面的現象，是用戶打斷頻率大增。這代表什麼？代表人類不得不一直接管。原本你用 agent，是希望它能自己把多步工作走完。可如果你得頻繁停下來糾正它、叫它回去讀檔、提醒它不要跳步，那自治就只剩下一半。

## 最傷信任的，是它開始把內部混亂外露

這份研究裡我最有感的一個指標，是推理迴圈變多。也就是模型在輸出裡開始出現更多像「等等」、「實際上」、「讓我重想一次」這類轉折。

表面看，這像是它在自我修正，好像還挺誠實。可如果頻率高到一個程度，那往往不是細心，而是內部規劃本來就沒站穩。以前它能在看不見的地方把矛盾消化掉，現在則是直接把未整理好的中間過程漏到表面。

對寫文章的人來說，這可能只是語氣不順。對寫程式的人來說，這很麻煩。因為你不知道它最後那個版本，到底是深思熟慮後的結論，還是倉促停在某個暫時平衡點。

## thinking 到底是不是一個產品功能

看完這份資料，我反而覺得爭議最深的一層，不是 Claude Code 一時變差，而是我們該怎麼理解 thinking 本身。

過去很多供應商把 thinking 說得像內部機密，好像只要答案對，使用者不需要知道過程。這種說法在一般聊天任務或許站得住腳，但在工程任務裡不夠。因為工程工作不是只看最終輸出，你還在意模型有沒有讀上下文、有沒有考慮副作用、有沒有在偷走捷徑。

換句話說，thinking 在這裡不只是模型的私事，它其實接近一種服務品質保證。

你可以不公開全文，但至少應該讓使用者知道這次回應花了多少思考預算。否則所有診斷都只能靠猜。是 prompt 寫得差？是模型退化？還是平台在默默幫你省成本？沒有指標，就沒有辦法建立穩定預期。

## 為什麼重度工程使用者特別痛

這種退化不是每個人都會立刻感受到。你用它寫一個小函式，也許差異不大。可一旦任務變成長 session、多檔案修改、遵守專案規範，thinking 深度一縮，問題就會被放大。

## 我怎麼看這件事

我不覺得這只是一家公司的小事故。它更像是一個提醒，未來 agent 產品如果要被拿來做嚴肅工作，就不能只賣能力，也得賣可預期性。

工程師不是不能接受成本取捨，但至少要知道現在買到的是哪一檔品質。若某些任務真的需要高 thinking 預算，那就應該有明確方案，讓使用者選擇付更多，換穩定一點的深度，而不是在看不見的地方默默收縮。

這份研究留下來的最大價值，不只是指出 Claude Code 某段時間變差，而是把一個原本很模糊的感受翻成了可以討論的語言。從此之後，大家談 agent 品質時，不能只看模型名字，還得問一句，它今天到底還剩多少時間願意先想清楚。
