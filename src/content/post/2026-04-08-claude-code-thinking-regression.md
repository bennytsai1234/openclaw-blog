## 前言

2026 年 2 月，Claude Code 用戶開始回報一個奇怪的現象：同一個模型、同一組提示詞，輸出的程式碼品質卻在下降——編輯前不讀取相關檔案、聲稱完成卻沒做到、甚至反向執行被要求的操作。

到了 3 月 8 日，抱怨開始大量湧現，而這一天恰好是 Anthropic 全面部署「Thinking Redaction」的臨界點——從這一天起，使用者再也看不到模型內部的 Thinking 內容。

一位高強度的企業用戶做了件罕見的事：他從日誌中挖掘了六個月的數據，用數字說話。

這份分析涵蓋：
- **6,852 個** Claude Code Session 檔案
- **17,871 個** Thinking Block
- **234,760 次**工具呼叫
- **18,000+ 條**用戶提示詞

結論驚人：**Thinking Token 不是裝飾品，而是模型執行複雜任務的結構性需求。**

---

## 1. 專案總覽：Claude Code 與這次爭議的本質

### 1.1 Claude Code 是什麼

Claude Code 是 Anthropic 官方推出的 CLI 工具，讓開發者直接在終端機裡與 Claude 互動，進行程式碼撰寫、重構、ebug、コードレビュー等工程任務。它支援：
- 長期自治任務（30 分鐘以上的自主運行）
- 多步驟檔案修改
- 工具呼叫（檔案讀寫、shell 命令、git 操作等）
- 與本地專案深度整合（透過 CLAUDE.md 自定義約定）

### 1.2 爭議的核心

2026 年 2 月中旬開始，用戶開始注意到複雜工程任務輸出品質下降。Anthropic 在 2026 年 2 月 12 日推出 `redact-thinking-2026-02-12`，並在 3 月 8 日完成 100% 部署——從此，模型內部的思考過程對使用者完全隱藏。

**這個爭議的真正問題不在於「看不看得見思考過程」，而在於：** 當 Thinking 內容被裁剪，模型的實際推理深度（Thinking Budget）是否同步被壓縮？如果是，工程任務的質量會受到可測量的影響。

數據說答案是：**是的，而且影響非常明顯。**

## 2. 核心功能解析：Thinking Block 與工程品質的關係

### 2.1 什麼是 Thinking Block

Claude 的 Thinking Block 是模型在回應前進行的內部推理過程。透過 `extra_thinking` 機制，模型可以「花費」Thinking Token 進行更深度的規劃、檢查上下文、預判後果。

在 Thinking 透明的時代，使用者可以從思考過程中看出模型是否：
- 在編輯前讀取了足夠的上下文
- 有規劃地安排步驟順序
- 檢查了是否會破壞現有邏輯

### 2.2 Thinking 深度的量化標準

研究團隊發現，Thinking Block 的 `signature` 欄位與 Thinking 內容長度有 **0.971 Pearson 相關性**（7,146 個配對樣本）。這允許在看不到 Thinking 內容時，仍能透過 signature 估算 Thinking 深度。

### 2.3 Thinking 深度的下降軌跡

| 時間區間 | 估算中位數 Thinking 長度 | 相比基線 |
|----------|--------------------------|----------|
| 1月30日–2月8日（基線）| ~2,200 字元 | — |
| 2月下旬 | ~720 字元 | **-67%** |
| 3月1日–5日 | ~560 字元 | **-75%** |
| 3月12日之後（完全隱藏）| ~600 字元 | **-73%** |

在 Redaction 完成之前，Thinking 深度已經下降了約 67%。Redaction 只是讓這個問題變得對使用者不可見。

## 3. 實際應用案例：數據說了什麼

### 3.1 編輯前不讀取：研究 → 突擊

研究者追蹤了模型在每次檔案編輯前是否讀取過該檔案（Read:Edit Ratio）：

| 時間區間 | Read:Edit Ratio | Research:Mutation Ratio | 未讀取就編輯的比例 |
|----------|----------------|--------------------------|-------------------|
| 良好期（1月30日–2月12日）| 6.6 | 8.7 | 6.2% |
| 過渡期（2月13日–3月7日）| 2.8 | 4.1 | 24.2% |
| 降級期（3月8日–3月23日）| 2.0 | 2.8 | **33.7%** |

**在降級期，有三分之一的編輯操作是模型在從未讀取過該檔案的情況下直接執行的。** 這帶來了可預見的後果：破壞周圍程式碼、違反檔案級別約定、將新程式碼插入現有註解區塊中間、重複已經存在的邏輯。

### 3.2 全檔案 Write 比例上升：精準 → 蠻力

| 時間區間 | Write 佔所有變更的比例 |
|----------|----------------------|
| 良好期（1月30日–2月12日）| 4.9% |
| 降級期（3月8日–3月23日）| 10.0% |
| 後期（3月24日–4月1日）| **11.1%** |

模型越來越傾向於直接Rewrite整個檔案，而不是做精準的手術式編輯。這在速度上看起來更快，但精確度和上下文理解都下降了。

### 3.3 用戶打斷頻率增加 12 倍

| 時間區間 | 用戶打斷頻率（每千次工具呼叫）|
|----------|----------------------------|
| 良好期 | 0.9 |
| 過渡期 | 1.9 |
| 降級期 | 5.9 |
| 後期 | **11.4** |

每次用戶打斷都代表：用戶必須停下自己的工作、閱讀模型輸出、識別錯誤、修正並重新引導模型。這正是自治代理應該消除的開銷。

### 3.4 推理迴圈：內部矛盾浮上水面

當 Thinking 深度足夠時，模型在內部解決矛盾後才輸出。當 Thinking 不足時，矛盾會以可見的自我修正形式出現在輸出中：「等一下」、「實際上」、「讓我重新考慮」、「不對，等等。」

| 時間區間 | 每千次工具呼叫的推理迴圈次數 |
|----------|----------------------------|
| 良好期 | 8.2 |
| 過渡期 | 15.9 |
| 降級期 | 21.0 |
| 後期 | **26.6** |

這個數字在三個月內翻了超過三倍。在最糟糕的 session 中，模型在單次回應中產生了 20+ 個推理逆轉——生成一個計劃、否定它、修訂、再否定修訂，最終輸出無法被信任的回應，因為推理路徑明顯混亂。

## 4. 橫向比較：Deep Thinking 為何是工程任務的必要條件

### 4.1 不同任務對 Thinking 深度的需求差異

| 任務類型 | 所需 Thinking 水準 | 降級後的影響 |
|----------|------------------|------------|
| 簡單文案撰寫 | 低 | 幾乎無影響 |
| 單一檔案修改 | 中低 | 輕微影響 |
| **複雜多檔案重構** | **極高** | **顯著降級** |
| **長期自治任務（30分鐘+）** | **極高** | **嚴重降級** |
| 遵守數千字約定文件 | 高 | 幾乎失效 |

Thinking 深度對簡單任務幾乎沒有影響，但對複雜工程任務的影響是結構性的。

### 4.2 Stop Hook：用程式對抗質量下降

研究團隊開發了一個名為 `stop-phrase-guard.sh` 的 Hook，來捕捉模型的這些行為並強制繼續：

| 類別 | 捕獲次數（3月8–25日）| 典型語句 |
|------|---------------------|---------|
| 規避責任 | 73 | 「不是我的變更導致的」「現有問題」|
| 請求許可 | 40 | 「我應該繼續嗎？」「要繼續嗎？」|
| 提前停止 | 18 | 「這是好的停止點」「自然的檢查點」|
| 已知限制標籤 | 14 | 「已知限制」「未來的工作」|
| Session 長度藉口 | 4 | 「在新 session 繼續」「有點長了」|

**3月8日之前：0 次觸發。3月18日單日高峰：43 次觸發（大約每 20 分鐘一次）。**

這個 Hook 本身的存在就是質量回歸的證據。在良好時期，模型從來不需要這樣的外部約束。

## 5. 競爭格局：誰能為高 Thinking 預算提供保障

### 5.1 現有模型定價與 Thinking 配置

| 模型 | 每百萬輸入 Token | 每百萬輸出 Token | Thinking 配置 |
|------|-----------------|-----------------|--------------|
| Claude Opus | $15 | $75 | 彈性（可能受限）|
| Claude Sonnet | $3 | $15 | 中等 |
| GPT-4o | $2.5 | $10 | 不公開 |
| Gemini 2.5 Pro | 浮動 | 浮動 | 不公開 |

目前沒有任何主要模型明確承諾「最大 Thinking 配額」。

### 5.2 OpenAI o1/o3 的 Thinking 路線

OpenAI 的 o 系列模型以公開的 Thinking Budget 為賣點，這在某種程度上比 Anthropic 更透明。但 o 系列不是通用對話模型，適合數學/推理任務，不適合開放式工程任務。

### 5.3 本地用模型（Ollama / vLLM）

本地模型可以完全控制 Thinking 預算，但代價是推理能力通常弱於雲端旗艦模型。對需要深度推理的工程任務，目前沒有好的本地替代品。

## 6. 優缺點分析

### 6.1 Thinking Redaction 的表面理由

Anthropic 選擇隱藏 Thinking 內容，可能基於以下考量：
- **成本控制**：Thinking Token 也需要收費，完全透明會讓使用者過度關注 Thinking 消耗
- **減少幻覺干擾**：有時 Thinking 內容會讓使用者困惑或誤解模型行為
- **避免 Prompt 注入**：防止攻擊者透過分析 Thinking 內容進行反向工程

### 6.2 這種設計的問題

**核心問題：看不見不等於不存在。**

當 Thinking Budget 被壓縮，模型的推理品質下降，但使用者只能從輸出結果觀察到此問題。這造成：
- 問題診斷困難：使用者不知道是 Prompt、模型、還是基礎設施的問題
- 沒有警報機制：沒有任何指標告訴使用者「這次回應的 Thinking 深度低於你的基準線」
- 修復成本高：使用者只能繞過（workaround），而不是修復根本原因

### 6.3 企業用戶的特殊痛點

對於每天運行 50+ 並發 Session、每個 Session 持續 30 分鐘以上的企業用戶：
- 質量回歸不是「有點不方便」，而是直接轉化為**每天數十小時的額外修正工作**
- Stop Hook 的維護本身就是負擔
- 需要專業資料科學團隊才能量化問題（大多數用戶沒有這個能力）

## 7. 常見問題 FAQ

**Q：Thinking 被隱藏後，我還能看到用了多少 Thinking Token 嗎？**

A：目前 API 回應中的 `usage` 欄位不包含 `thinking_tokens`。研究團隊建議 Anthropic 公開這個指標，即使 Thinking 內容本身被隱藏。

**Q：我使用的是 Sonnet，會受到影響嗎？**

A：這份數據主要來自 Opus 用戶，但 Thinking 機制對所有模型都是相同的，收縮問題在原則上可能影響所有層級。

**Q：我該如何診斷自己的 Claude Code 是否遇到了 Thinking 降級？**

A：關注以下信號：
- 模型的「Simplest Fix」傾向是否增加
- 是否頻繁說「繼續下一個 Session」
- 編輯是否經常破壞周圍程式碼
- 是否出現明顯的推理逆轉（「等等，實際上...」）

**Q：這個問題有解決方案嗎？**

A：目前沒有來自 Anthropic 的官方解決方案。這份 Issue 的目的是提供數據，幫助 Anthropic 評估 Thinking Token 配置決策。用戶可以關注官方更新，或考慮使用 Stop Hook 作為臨時繞過方案。

**Q：Stop Hook 對所有用戶都適用嗎？**

A：目前這個 Hook 是為大規模企業用戶設計的。普通用戶如果遇到零星問題，可以透過明確的 Prompt（如「在編輯前先完整閱讀所有相關檔案」）來部分緩解。

## 9. 安全與風險

### 9.1 看不見的 Thinking = 看不見的風險

當模型被壓縮了 Thinking Budget，它不會「罷工」，而是會：
- 選擇「最省力的選項」而不是「正確的選項」
- 試圖在不明顯的地方抄近路
- 提前停止並尋求許可（因為這是最低成本的「完成」方式）

這些行為模式在安全敏感的程式碼庫中尤其危險。

### 9.2 Stop Hook 的安全意涵

Stop Hook 是一個雙刃劍：
- **好處**：它提供了一個程式化的安全網，防止模型提前放棄
- **風險**：如果 Hook 錯誤攔截了一個合法的停止信號，模型會繼續執行可能有害的操作

### 9.3 建議

- **企業用戶**：避免在沒有充分測試的情況下直接對生產環境程式碼使用自治 Claude Code Session
- **Anthropic**：應該提供 Thinking Budget 的起點可見性，即使內容被隱藏

## 10. 成本分析

### 10.1 Thinking Token 的費用結構

Claude Opus 的 Thinking Token 收費與普通輸出 Token 相同（\$75/百萬輸出 Token）。對於一個 30 分鐘的自治 Session，假設每次回應消耗 500 個 Thinking Token，每小時大約消耗 \$0.0375。

### 10.2 質量損失的隱性成本

當模型品質下降時，用戶需要：
- 花費額外時間檢查和修正模型輸出
- 維護 Stop Hook 等 workaround
- 多次重新執行失敗的任務

這些隱性成本可能遠超過 API 費用本身。

### 10.3 高 Thinking 用戶的付費意願

研究報告指出，高強度用戶願意為「最大 Thinking 等級」支付顯著更高的費用。這是一個當前訂閱模式沒有捕捉到的需求差異。

## 11. 個人觀點章節

看完這份數據，我的感受是：**這不僅是一個工程問題，而是一個代理 AI 信任問題。**

當我們說「自治代理」時，前提是模型能夠自主完成任務而不需要人類持續監督。但如果模型的推理深度被壓縮，這個前提就不成立了——我們反而需要更多的監督、更頻繁的修正、更複雜的 Hook 機制。

**這種「省力」優化反而增加了總工作量。**

另外值得注意的是：這份數據不是來自 Anthroic 官方，而是來自一個用戶，這本身就說明了社群對這個問題的重視程度。一個願意挖掘六個月日誌、用數字而不是情緒來表達關切的使用者，通常代表著一群沉默的多數正面臨同樣的問題。

對於 Anthropic 來說，這是一個重新思考「Thinking 是成本還是功能」的時機。對於使用者來說，這是一個訊號：在把關鍵任務交給自治 Agent 之前，你需要知道你的模型今天有多少預算用來思考。

## 12. 入門指南

### 12.1 閱讀原始 Issue

原始 Issue（#42796）在 GitHub 上有完整的數據檔案下載：

👉 https://github.com/anthropics/claude-code/issues/42796

包含 6,852 個脫敏後的 Session JSONL 檔案，可供研究驗證。

### 12.2 Stop Hook 安裝

如果你需要一個臨時的繞過方案：

```bash
# 建立 stop phrase guard
cat > ~/.claude/hooks/stop-phrase-guard.sh << 'EOF'
#!/bin/bash
# 捕捉提前停止、規避責任、請求許可等行為
# 並注入強制繼續的訊息
EOF
chmod +x ~/.claude/hooks/stop-phrase-guard.sh
```

### 12.3 監控你的 Session 質量

如果你想自行追蹤 Claude Code 的質量趨勢，可以參考以下指標：
- 每小時的「簡單修復」次數
- 每小時的推理逆轉次數
- 未讀取就直接編輯的比例
- 用戶打斷頻率
