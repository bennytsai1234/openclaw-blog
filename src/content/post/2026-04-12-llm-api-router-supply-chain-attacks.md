很多人接第三方 LLM API 路由器時，心裡想的是方便。統一帳務、切模型便宜、容錯也比較好，出事時還能快速換上游。這套想法本身沒錯，但它藏著一個很少被正面談的前提，你把流量先交給了別人。這篇研究最刺耳的地方就在這裡，它提醒我們，路由器不是單純水管，而是握有內容與行為改寫權的中介。

## 為什麼這個問題以前沒被當大事？

因為大家太習慣把 router 想成 infra。只要名稱叫路由器、代理層、gateway，工程師就容易直覺把它放在「傳輸優化」那一欄，好像它只是幫你挑模型、轉 URL、補重試。可實際上，當客戶端把 endpoint 指向路由器時，TLS 連線就是在那裡終止。之後它再怎麼往上游送，對使用者而言都已經是第二段旅程。

這代表什麼？代表它理論上就能看內容、改內容，也能重組工具呼叫。不是先攻破 HTTPS 才辦得到，而是系統設計一開始就把這個權力交給它了。

## 真正危險的點在哪裡？

最危險的不是它能看到 prompt，而是它能改 agent 將要執行的東西。

論文提到的 payload injection 很值得警惕。今天 agent 收到模型回傳的 tool call，開發者常常預設那是模型和工具之間的正常橋接。但如果中間路由器把某段 JSON、某條 shell 指令、某個安裝命令換掉，客戶端未必看得出來。表面上你仍在執行「模型給的動作」，實際上可能已經被插入惡意行為。

這種風險在 agent 場景比一般聊天更麻煩，因為輸出不再只是文字，而是可能直接通往檔案系統、shell、付款流程與內網服務。換句話說，路由器一旦惡意，不只是偷看，而是能動手。

## 開源與商業路由器，誰比較安全？

老實說，名稱不是重點，供應鏈路徑才是。很多商業服務其實也是從 LiteLLM、new-api、one-api 這類開源模板長出來的，再加上一些上游轉發與客製層。問題在於，中間可能不只一層。你以為自己連的是某個整合服務，實際路徑可能還會再經過另一個聚合平台，最後才到模型供應商。

只要其中一層被動過手腳，全鏈就有風險。這也是為什麼這篇研究讓人不安，因為它指出惡意案例不是理論推演，而是市面上已經抓得到樣本。

## 那我們是不是不該用 router？

也不能這麼簡單下結論。路由器的確有實際價值，尤其在成本管理、故障切換、多供應商整合上，它幫很多團隊省下大量工程時間。問題不是能不能用，而是不能再把它當無害基礎設施。

如果你的 agent 會執行工具、下指令、寫檔案，那 router 就應該被當成高權限供應鏈節點來看。該做的不是盲信，而是驗證。像是工具呼叫簽章、回傳結構校驗、命令白名單、敏感操作二次確認、最小權限隔離，這些以前可能被當成進階防護，現在應該變成基本配備。

## 這篇研究真正改變了什麼？

它改變的是大家看待信任邊界的方式。過去很多 agent 架構圖把模型供應商畫成主要風險來源，卻把中間的 router 畫得很淡。現在我們知道，真正危險的地方可能不是最遠的那一端，而是你為了方便主動接上的那一層。

AI 系統發展得太快，很多團隊先求能跑，再補治理。這沒有什麼好羞愧，大家都是這樣往前推的。但這篇論文像在半夜把燈打開，讓人看見桌上其實多了幾條沒注意到的線。它不會讓 router 從此消失，卻會逼成熟團隊重新問一次，那些替我們省事的中介，到底被我們給了多少不該那麼輕易交出去的權力。
