在企業級產品裡用 LLM 跑問答功能，工程師遲早會碰到一種不是 bug、但比 bug 還難搞的狀況：模型生命週期到了。Claude 3 Haiku 宣布退役，GPT-4o 被下架，Qwen-32B 突然說要換架構——這些都不是技術錯誤，而是商務決策。當支撐產品的基底模型要換掉，怎麼知道新模型到底行不行？

這件事做起來比看起來複雜太多。

## 問題不在於找到更好的模型

多數人直覺反應是：那找一個評測分數更高的模型不就得了？Benchmark 刷上去，遷移就結束了。這個思路在研究論文裡常見，但在實際企業產品環境中碰壁，因為你其實根本不知道自己在測什麼。

拿問答系統來說，系統輸出品質取決於三個維度：答案是否正確、系統何時選擇說「我不知道」（IDK）、以及答案的文風是否符合商業場景的規範。這三件事各自獨立在不同的方面影響使用者體驗，而多數公開的自動化評測指標——BLEU、ROUGE，甚至 RAGAS 的 faithfulness 與 relevance——在面對 IDK 這件事上根本沒有適當的處理邏輯。他們會把 IDK 回應直接當作「錯誤」計分，但對企業級客服系統來說，正確地說「我不知道」其實是優於胡亂給出一個似是而非的答案的。

問題複雜度還不只如此。這套系統支撐的產品每個月處理 530 萬次對話，橫跨六個全球區域，支援十幾種語言，而且這些模型是由第三方雲端平台托管的商業模型——不是你可以自由改參數的那種。遷移的時間窗口很短，舊模型說下架就是下架，但新模型候選名單裡，每一個都要同時通過合規審查、資料落地限制、版權約束與成本上限。這些條件會直接把多數「分數很漂亮」的模型刷掉，不是技術問題，是商務限制。

所以真正的問題不是「哪個模型更好」，而是「在合規、成本、區域、延遲全部卡死的約束下，如何有信心地選出一個不會讓產品變差的替代方案」。

## 舊做法早就碰到牆，只是大家沒有明講

在 LLM 還沒那麼普及的時代，模型遷移這件事不常見。但這幾年商業 LLM 的更迭節奏已經變成大約每十二個月一次，幾乎每個建立在商業模型上的企業產品每一年都要面對一次遷移。業界對這個問題的標準處理方式，其實說穿了就是拿公開 benchmark 選模型，然後快速測一下，沒有問題就上線。

這套做法為什麼行不通？第一個原因是公開測試集和真實業務場景之間的鴻溝。HotpotQA 和 SQuAD 是很好的學術資料集，但它們的問題類型和文件風格跟企業客服對話的實際資料分布差異很大。用這些測試集評出來的結論，經常跟實際產品表現對不上。第二個原因是提示詞耦合問題。同一套 prompt 在 Model A 上測試結果漂亮，換到 Model B 上可能輸出格式整個跑掉，或者風格突然變得很機器人。一套 prompt 不能無痛地在不同模型間迁移，這是工程團隊最常低估的隐性成本。

第三個原因——也是這篇論文最核心的切入點——是自動化評測指標本身並不可靠。現有指標對正確性的判斷有偏差，而且這種偏差是系統性的，不會因為樣本數變大而消失。用這些指標做模型之間的比較，結論可能是錯誤的。

## Bayesian 統計方法怎麼繞過這個問題

Verint 團隊提出的框架核心，是用 Bayesian 統計把這個問題翻轉過來。他們不再試圖找到一個「完美正確的自動化指標」，而是承認指標會有偏差，然後用很少量的人工標注來校正這個偏差。

具體做法是這樣的：先用手動評測建立一個小子集（約 146 筆，隨機從 HotpotQA 和內部測試集抽出），由三名人員共同對這些答案做正確性評判，並記錄自動化指標在這些案例上的預測與人工判斷之間的差距。這個差距構成一組 confusion matrix：哪些案例被指標錯誤地標記為正確，哪些被錯誤地標記為錯誤。

拿到這組 confusion matrix 之後，就可以用 Bayesian 方法建立一個帶有先驗的統計模型，估算每個自動化指標的真陽率（TPR）和假陽率（FPR）。這個估算本身帶有置信區間——樣本數越小，置信區間越寬，結論越保守。當實際比較候選模型與基準模型時，報告的不是一個點估計值，而是一個範圍：「新模型正確率比 Haiku 高 2% 到 8%，置信度 90%」。這個區間裡如果包含負值，就說明證據不足，不能自信地宣稱新模型更好。

用這個方法，團隊最終測試了 11 個候選模型，通過內部合規審查的 Gemma 3 27B、Claude 4.5 Haiku、Claude 3.5 Sonnet、AWS Nova Micro、Lite、Lite 2、Pro、OpenAI GPT-OSS 20B 和 120B、Qwen3-32B 和 Qwen3-235B。首先用輸出格式服從性刷掉兩個 GPT-OSS 模型（它們生成的 XML 不符合規範），Qwen3-32B 也遇到類似問題但通過在 prompt 中加入格式範例快速修復。接著用 new_correctness 指標（新答案是否與完整上下文一致，而不僅僅是跟標準答案字面一致）加上 Bayesian 置信區間比較，刷掉 Nova Micro、Nova Lite 和 Gemma 3 27B——它們的置信區間都包含負值，無法自信排除效能退化。

最後剩下的五個模型在風格和延遲維度上進一步分化：Claude 3.5 Sonnet 延遲顯著偏高（1.73 秒相對基準的 1.19 秒），Qwen3-32B 的推理模式會讓風格退化（bad style 從 2.1% 上升到 7.5%）。最終方案是 Nova 2 Lite 負責文件處理場景，Qwen3-32B（關閉推理模式）作為主要語言模型。

## 提示詞遷移：三種策略的實際體驗

模型選定之後，還有一件事不能跳過：提示詞遷移。同一套給 Claude 3 Haiku 設計的 prompt，到了 Qwen3-32B 上不一定有同樣效果。論文測試了三種提示詞適配策略。

第一種是手動適配——根據模型官方文件的建議做法，調整關鍵指令的大小寫、加入更具體的上下文範例。第二種是使用 Amazon Bedrock 的自動提示詞優化工具。第三種是 MIPROv2 之類的自動提示詞優化框架。

結果讓人有點意外：自動提示詞重寫還沒有達到可以完全依賴的水準。Jahani 等人 2026 年的研究也指出，自動提示詞改寫在模型切換場景下還不夠可靠，最好還是需要人工介入做判斷。

## 這件事的代價與被高估的地方

整個框架的價值在於提供了一個系統性的、可重複的遷移流程，讓企業不需要每次模型退役都要從零開始重新設計評測方法。Bayesian 置信區間的引入讓決策從「感覺新模型更好」變成「有資料支援的結論」，這對需要向上交代的工程團隊來說是重要的。

但也要說清楚這件事的代價：光是手動標注 146 筆案例、對三個測試集建立 confusion matrix，就已經是相當大的人力投入。論文裡提到光是讓三名標注人員在marginal cases上達成共識就花了相當時間——這說明品質評估這件事本身就沒有辦法完全自動化，人工判斷的環節是躲不掉的。

另外容易被高估的部分，是這套框架的可遷移性。論文的測試集是商業客服問答系統，輸出格式要求是結構化 XML，而且商業約束條件（多語言、區域合規、成本上限）都與特定企業高度綁定。框架的原則（少量人工標注校正自動化指標 + Bayesian 置信區間決策）在其他場景下應該也適用，但具體的指標選用、測試集設計、以及閾值設定都需要根據實際業務重新調整。

## 讀到這裡，你應該判斷的事

如果你的產品是建立在商業 LLM API 上的問答或對話系統，而且你現在用的是單一模型、單一 prompt、沒有系統性的模型替換預案，這篇論文值得你認真評估。530 萬月互動量的系統可以用 Bayesian 方法把决策错误率压到可接受范围，你的系统不一定需要同样复杂的流程，但你需要知道的是：没有系统性的评估框架，模型迁移就是在赌。

如果你的团队現在正在經歷 LLM 退役焦慮，或者正在手動凭感觉选新模型，這篇論文提供了一個思考框架：与其追求完美的自动化指标，不如承认指标的局限性，然后用统计方法量化不确定性，让决策有据可查。

这不是一篇告诉你「用一个新模型就好」的论文。它告诉你的是：当你不得不换的时候，怎么换得有信心。
