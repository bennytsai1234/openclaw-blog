花了很長時間把一個 AI Agent 部署到實際工作流程裡，結果發現瓶頸根本不在模型本身——而在鞍具（harness）上。這大概是 2026 年上半年最讓工程團隊感到挫折的事情之一：你花了三週微調提示詞、重構工具呼叫邏輯、補強評估函式，等 model 換了一代，突然全部要從頭來過。模型厲害，但鞍具是跟著人走的，人走了、經驗帶不走。

這就是 Haebin Seong、Li Yin、Haoran Zhang 三位作者在四月提出的研究的核心處境。他們說：**鞍具工程（harness engineering）才是真正的瓶頸，而這件事遲早要自動化。**論文標題乾脆就叫《The Last Harness You'll Ever Build》，那個「你最後一次親手打造的鞍具」，現在就來把它讀完。

## 一個大家都碰過的牆

在進入論文的方法之前，先把問題看清楚。你或許用過 Claude Code 或 OpenAI Codex，你或許知道這些工具厲害的地方並不是模型本身——而是它們被包裝成 Agent 的那一層「鞍具」。鞍具決定了 agent 能看到什麼、能做什麼、以及它的表現如何被驗證。Claude Code 的檔案系統隔離、Codex 的多輪回顧機制，都是鞍具的一部分，不是模型的能力。

問題在於：打造一套好鞍具，需要深度領域知識，還需要大量迭代。Lopopolo 在 2026 年的研究裡描述過他建一個針對陌生程式碼庫的 agent 需要什麼：客製化的 linter、程式庫本地的可觀測性堆疊、Chrome DevTools 整合、以及結構化的文件層級——全部手工打造。Rajasekaran 另一篇同年的研究也報告過，設計一個好的評估器需要好幾輪提示詞校正、為主觀設計品質建立四級評分標準、還要架一個 planner-generator-evaluator 三 agent 架構。每一個新的任務領域，都意味著從零開始的專家人力投入。

這就是論文所指的那道牆：鞍具改善了 agent，但**改善鞍具本身**仍然需要人在每個領域裡持續投入專家時間。這不是 scale 得起來的事情。

## 兩個疊起來的進化迴圈

作者的回應是一個兩層框架，核心概念是：**鞍具不只是一個給定任務的設計物件，它本身就是一個可以被優化的系統。**

第一層叫 Harness Evolution Loop，針對單一任務優化一個 worker agent 的鞍具。迴圈裡有三個角色：

**Worker Agent** W_ℋ 是被優化的目標，它執行任務、產生執行軌跡。ℋ 代表這次優化的鞍具配置，初始可以是完全未調校的通用框架。

**Evaluator Agent** V 是對抗式的審查者。它讀取 worker 的執行軌跡和任務定義，輸出結構化的診斷報告與分數。它的功能分成四層：狀態驗證（確認 worker 實際觀察到的環境狀態跟它聲稱的一致，檢測 hallucinated 觀察）、標準檢查（對每個成功條件給出 pass/fail）、效能審計（區分 LLM 推理延遲和工具呼叫延遲），以及評分（任務完成優先，執行時間作 tiebreaker）。

**Evolution Agent** E 是整個迴圈的演化驅動。它接收完整的進化歷史——所有嘗試過的鞍具變體、它們的評估報告、分數、以及每一次改進或回退的結果——以及目前表現最好的鞍具，然後產生下一個版本的鞍具。它像一個資深工程師那樣聚合診斷資訊、把失敗模式分類、並對 worker 的鞍具做出有根據的修改。

整個內層迴圈迭代 K 次，每次都從最好的鞍具出發，試圖演化出更好的版本，最後輸出這次任務的最佳鞍具 ℋ^(best)。

第二層叫 Meta-Evolution Loop，這是作者真正想強調的部分：**鞍具進化迴圈本身也是一個鞍具，現在我們連這個鞍具的設計都自動化。**定義 Λ = (W_ℋ, ℋ^(0), V, E)，也就是 worker agent、初始鞍具、evaluator 和 evolution agent 的組合，這四個東西湊在一起定義了「如何演化鞍具」這件事本身。而 Λ 本身也是可以被修改的——包括 evaluator 的提示詞、evolution agent 的策略、評分函式結構、以及迭代次數與終止條件這些超參數。

Meta-Evolution Loop 在多個不同任務上運行第一層迴圈，評估目前的 Λ 在這些任務上的適應速度與最終表現，然後讓一個 Meta-Evolution Agent 去修改 Λ 本身，目標是找到一個 Λ^(best)——這個進化協議能讓內層迴圈在任何新任務上都快速收斂到高效鞍具。

形式上，這對應到 meta-learning 框架：內層是在固定 Λ 下適應單一任務，外層是在跨任務學習最優的 Λ。

## 為什麼這個方向值得認真對待

這篇論文不是第一個討論 prompt 自動優化的研究。LLM-AutoDiff（Yin and Wang, 2025）這類方法可以對個別 components 做 tuning，但從未試圖處理整個鞍具——工具、 orchestration 邏輯、基礎設施、以及它們之間的互動。這個框架的野心更大：它把個體任務上的鞍具工程，和跨任務的進化協議工程，放在同一個優化問題裡。

從工程視角看，這個框架試圖處理的問題是真實的。當你把 Claude Code 部署到一個新的程式碼庫時，你不是在訓練一個新模型，你是在建一個讓現有模型能有效運作的環境。問題在於這個環境的建設目前高度依賴專家判斷，而作者指出這件事可以疊代自動化。

當然，這裡有些真實的限制值得關注。首先是計算成本：每個任務的內層迴圈需要 K 次 worker執行+評估+演化，在 meta-evolution 層級還要跨 N 個任務重複這個過程，總成本會是很可觀的。其次，evaluator 本身的能力決定了進化方向的品質——如果 evaluator 有盲點，整個迴圈都會被帶偏。最後，論文目前是在受控環境下驗證概念，延伸到開放式的企業 web applications 會遇到什麼，真實環境的狀態多樣性是否會讓 evaluator 失效，這些都還需要更多實驗數據。

## 閒聊式的判斷

這篇論文提出來的框架在概念層級是乾淨的——它把「鞍具工程」這件事本身形式化成一個可以優化的物件，然後用雙層進化迴圈去解決它。這個思路如果成立，意味著未來部署 AI Agent 到新領域時，原則上不需要從零開始建鞍具，只需要一個經過 meta-evolution 調校的通用進化協議，讓它在目標任務上自動收斂。

這件事的價值不在於「省了工程師三天時間」，而在於：**它把鞍具從一個需要人類專家累積經驗才能建立的東西，變成一個可以用資料驅動、跨任務复用的系統**。這是從專家系統到學習系統的轉變，方向上值得持續關注。

但也不要用它來否定 human-in-the-loop 的價值。鞍具最後仍然是由人設計的進化協議在指導方向，而且真實部署場景裡的 failure mode 通常不是「鞍具收斂不夠快」，而是「鞍具在錯誤的方向上收斂得很穩定」。在相對高風險的應用場景裡，完全自動化仍然需要驗證護欄。

對工程師來說，這篇論文最有意思的地方或許不是那個兩層框架，而是背後那個 observation：**當我們說一個 AI agent 「不好用」的時候，問題往往不是模型不夠聰明，而是我們沒有把它的能力封裝进一個足夠好的執行環境。**把這個環境的設計從手工變成自動化，是 2026 年這個領域最值得投入的方向之一。
