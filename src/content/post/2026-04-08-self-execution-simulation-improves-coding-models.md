寫程式的人都知道，真正把 bug 找出來的時刻，常常不是按下執行鍵之後，而是在那之前。你盯著一段迴圈，看變數怎麼流，想像某個分支被走到，忽然意識到，啊，這裡會炸。這種能力不華麗，但它是工程師的基本功。

現在的程式模型其實很少有這個本事。它們會生成 code，也會模仿解題語氣，甚至能把演算法講得頭頭是道。可是一旦問它，這段程式在這個 input 下到底會輸出什麼，它往往還是在猜。原因很單純，模型平常學到的是文字分布，不是執行過程。

這篇 **Self-Execution Simulation Improves Coding Models** 論文，想補上的就是這塊缺口。作者不是只讓模型學會寫程式，而是讓它學會在回答之前，先模擬自己剛寫出來的程式會怎麼跑。

## 如果模型只會寫，不會跑，問題會卡在哪裡

過去幾年，Code LLM 的進步很快。從補全、重構，到競程解題，很多 benchmark 都往上衝。但一個根本限制一直沒消失，模型對程式的理解常停在字面層。它知道哪種寫法常見，知道哪種模板配哪種題型，卻不一定真的掌握執行時的狀態變化。

這會導致一種很熟悉的錯誤。程式看起來合理，甚至註解也寫得像模像樣，但變數在某一步就偏了，分支沒處理全，或索引在邊界上少一格。人類工程師看到這種程式，會下意識 trace 幾輪。模型以前沒有這個習慣。

作者的想法很直白，既然實際執行程式很貴，那能不能先訓練模型有一個「腦內執行器」？

## 這篇論文怎麼教模型這件事

方法分成兩段。

第一段是做資料。他們收集大量 Python 函式，真正執行，錄下每一步的狀態變化，再把這些原始 trace 轉成自然語言敘述。這批資料被稱為 **NLEX**。

這個設計很聰明。研究團隊沒有硬逼模型吃結構化的中間狀態，而是把執行過程翻成模型更熟悉的敘事形式。

第二段才是把這個能力接到解題任務上。模型先生成候選程式，再根據 public test 去模擬輸出，最後選出最可能正確的那份答案。論文裡把它做成 best@k 的自我驗證流程，也做了多輪的 self-RLEF scaffold，讓模型能在模擬失敗後嘗試修正。

說白一點，就是從「會寫題目」進化成「會寫，還會先自己檢查一下」。

## 效果是真的有，重點是有多可信

論文裡最好看的數字來自 CruxEval-O。Qwen2.5-3B 與 7B 經過 NLEX 訓練後，對程式輸出的預測能力明顯上升，幅度不小。到了 competitive programming 場景，self-verification 也帶來 2 到 8 個百分點的 pass rate 提升。

這裡最值得注意的，不只是最終通過率，而是模擬結果和真實執行的落差沒有想像中大。

這會讓整個 agent pipeline 的設計開始改變。以前你如果想驗證十份候選程式，可能得真的跑十次。未來更合理的做法，是先讓模型腦內模擬，把明顯不會過的答案篩掉。

## 這跟「真的跑程式」差在哪裡

當然，模擬不是執行。

一旦程式涉及檔案系統、網路 I/O、系統呼叫、隨機性，或一堆隱晦的 side effect，模型的腦內世界就會開始失真。競程題目適合這套方法，是因為輸入輸出邊界清楚，規格相對封閉。你給定 input，它就該吐出可比較的 stdout。現實專案不是這樣。

另一個風險是「錯上加錯」。如果模型寫出錯誤程式，又錯誤地模擬出一個看似合理的輸出，它就會對自己產生虛假的信心。論文裡也看得到這件事，所以他們特別指出，單靠 scaffold 不夠。

這也是我覺得這篇論文誠實的地方。它不是在賣一個萬用框架，而是清楚說，這是一種專門能力。

## 我比較在意的，是它把 code 從文字拉回到狀態機

這篇研究真正重要的地方，不只是分數提升，而是它提醒大家，程式碼不是一般文本。你可以把它印成字串，但它的核心其實是狀態轉移。變數怎麼變，控制流怎麼走，才是程式真正的骨架。

很多 Code LLM 之所以在簡單 benchmark 表現不錯，是因為題目和答案之間有很強的語料模式可以抓。可一旦要求它真的理解某段程式會怎麼跑，模式匹配就開始不夠用了。Self-Execution Simulation 做的事情，就是逼模型補上這段語意層。

## 我的判斷

我會把這篇論文看成一個很合理、也很務實的方向。它沒有承諾模型突然學會所有程式語意，但它證明了一件值得高興的事，執行模擬這種能力不是空想，真的可以靠資料和訓練長出來。

短期內，它最適合的地方還是競程、程式評測平台、受控的 code generation 流程。即使先不談更遠的應用，光是看到模型開始學會「先在腦中跑一遍」，我就覺得這方向很對。
