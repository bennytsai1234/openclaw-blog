如果你曾經嘗試在自己的筆電或邊緣設備上部署一個 7B 或 8B 規模的模型，你應該很快就會發現一個令人沮喪的事實：限制你的往往不是模型權重（Weights）的大小，而是那該死的 KV Cache。

這是一個非常弔詭的現象。我們稱 LLM 為「壓縮機器」，因為它們能將整個互聯網的知識壓縮進數十億個參數中；但在執行階段，它們卻展現出極其低效的記憶體行為。以 Llama-2-7B 為例，一個簡單的整數 token 在輸入時幾乎不佔空間，但為了在 autoregressive 生成過程中避免重複計算，模型必須在每一層都為這個 token 保存 Key 和 Value 向量。

結果就是，一個 token 在 Llama-2-7B 裡的 KV Cache 足跡大約是 512 KB。如果你想處理一本書的上下文，KV Cache 佔用的記憶體甚至可能超過模型本身的權重。這就是所謂的「KV Cache 牆」——它直接決定了你的最大 batch size 和上下文長度，也成了推理成本最核心的痛點。

## 為什麼「刪除 token」這條路走不通？

面對這個壓力，工業界的第一反應通常是：既然記憶體不夠，那就把不重要的 token 刪掉。

這就是目前主流的「時間軸刪除」（Temporal Eviction）路線。像 H2O 或 SnapKV 這類方案，試圖找出那些貢獻度低的 token 並將其從快取中剔除。邏輯很直觀：既然模型在注意力機制中只關注少數幾個關鍵點，那我只要保留這些「注意力沉積」（Attention Sinks）和最近的 token 即可。

但這種做法有一個致命的缺陷：**查詢依賴性（Query-dependency）**。

一個 token 在處理第一個問題時可能被判定為「不重要」而被刪除，但當用戶接著問第二個問題時，那個被刪掉的 token 可能恰恰是回答關鍵。這導致模型必須在「冒著幻覺風險刪除」與「承受高昂計算開銷重新計算（Re-compute）」之間做選擇。而且，為了決定誰該被刪除，系統還得額外跑一套評分邏輯，這本身就增加了延遲。

當我們在時間軸上打轉時，Apple 的研究團隊提出了一個不同的視角：既然時間軸太複雜，為什麼我們不看看「深度」方向？

## 深度冗餘：我們真的需要每一層都有快取嗎？

目前的 Transformer 解碼器結構非常僵硬：每一層（Layer）都有自己獨立的 Key 和 Value 投影矩陣，因此每一層都必須維護一套專屬的 KV Cache。

但事實上，模型內部的表徵（Representations）存在極高的跨層冗餘。簡單來說，第 20 層和第 21 層看到的資訊可能差不多。如果我們能讓第 21 層直接「借用」第 20 層的快取，我們就能直接省掉一整層的記憶體空間。

理論上這很美好，但在實踐中卻極其困難。如果你直接強迫一個預訓練好的模型去共享快取，性能會崩潰，因為每一層的權重是針對自己的特定特徵空間訓練的，它們並不兼容。而如果你想設計一個「共享快取」的架構，通常需要額外的編碼器（Encoder）來統一表徵，或者在預填充（Pre-fill）階段跑多次前向傳播來逼近正確值，這會讓首 token 延遲（TTFT）飆升。

Apple 提出的方案——隨機跨層注意力（Random Cross-Layer Attention, R-CLA），走了一條完全不同的路徑：**用隨機性來強迫模型學習「適應」**。

## R-CLA：在混沌中學習魯棒性

R-CLA 的核心思想不是在推理時去尋找最優共享路徑，而是在訓練（或微調）階段就讓模型習慣「快取出錯」。

具體做法非常簡單且暴力：在訓練的每一次前向傳播中，對於任何一個層 $l$，模型會以機率 $p$ 進行正常的自注意力計算（使用自己的 KV Cache），但會以 $1-p$ 的機率被強迫去讀取某個隨機選定的前序層 $l'$ 的 KV Cache。

你可以把它想像成一種「結構化的隨機丟包」。模型在訓練過程中會發現，它不能完全依賴於本層那個精準的特徵空間，而必須學會從之前層級的通用語義表徵中提取資訊。

這種 stochastic（隨機）的訓練過程產生了一個神奇的效果：模型變得對「深度共享策略」極其魯棒。

到了部署階段，開發者可以根據硬件限制，自由定義一種快取共享策略 $\mathcal{S}$。例如，如果你在高端 H100 集群上運行，你可以保留 100% 的快取以追求極限精度；如果你在記憶體極小的邊緣設備上運行，你可以設定每 4 層才存一個快取，中間的 3 層全部強制復用最近的快取。

因為模型在訓練時已經見過各種隨機的復用模式，它能夠在不同的保留率下維持相對穩定的性能，而不需要為每種硬件配置重新訓練一個模型。

## 從記憶體削減到正則化：一個意外的收穫

在 Apple 的實驗中，R-CLA 不僅實現了顯著的記憶體削減（在丟棄 50% 到 75% 的層級快取後，性能依然能維持在可接受範圍），還帶來了一個意外的副作用：**正則化效應**。

對於大型模型而言，這種隨機的跨層路由實際上起到了一種 Dropout 的作用。它防止了模型過度依賴於特定層的過擬合，迫使信息在深度方向上分佈得更均勻。結果是，在某些數據受限的場景下，經過 R-CLA 訓練的模型在全快取模式下的表現，甚至比原始的基線模型還要好。

這意味著我們不需要在「性能」和「記憶體」之間做絕對的零和交易，而是可以透過引入適度的隨機性，在降低成本的同時提升模型的泛化能力。

## 結語：從「剔除 token」到「重構深度」

長期以來，我們對 KV Cache 的優化一直集中在「時間軸」上——試圖通過複雜的算法去猜測哪些 token 是廢物。但 Apple 的這項研究提醒我們，深度維度提供了一個更穩健且正交的優化方向。

R-CLA 的價值不在於它提供了一個完美的壓縮比，而是在於它打破了模型權重與快取結構之間的強耦合。它將「記憶體預算」從一個硬性的架構限制，變成了一個可以在部署時動態調整的參數。

對於開發者來說，這意味著未來我們可能不再需要為「手機版」和「伺服器版」準備不同的量化或裁剪模型。同一個 R-CLA 模型，可以在雲端全速運行，也可以在你的設備上以「極簡快取模式」高效運作。

當我們不再試圖去猜測哪些 token 該被刪除，而是讓模型學會如何高效地在深度方向上共享資訊時，我們才真正開始在對抗那堵 KV Cache 牆。
