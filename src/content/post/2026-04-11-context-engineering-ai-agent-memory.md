很多 AI Agent demo 都有同一種魔術感。

剛開始的幾分鐘，它什麼都會。能改文案、查資料、拆任務、回覆得像真的懂你。可一旦任務變長，對話多幾輪，接進真實工具和真實資料後，它忽然像換了一個人。先前講過的事忘了，工具用錯，檢索抓錯文件，然後很自然地胡亂補完。

大家常把這種現象叫做失憶，好像問題出在模型記性不好。其實更接近的說法是，你從來沒有替它把記憶系統整理好。

Context Engineering 這個概念，之所以最近越來越重要，就是因為工程師終於慢慢承認，AI Agent 的穩定度常常不是輸在模型參數，而是輸在上下文怎麼進場。

## 不是裝更多資訊就會更聰明

很多團隊做 agent 的第一反應都很直觀，既然模型需要脈絡，那就塞更多脈絡。把歷史對話放進去，把知識庫抓進去，把工具說明列進去，把使用者偏好也一起帶著。最後 prompt 長到像搬家清單，卻還是沒比較可靠。

原因很簡單，context window 不是倉庫，比較像工作桌。你把所有東西都堆上去，模型不是因此視野更完整，而是更難分辨現在該看哪一份資料。

這也是 context engineering 跟一般 prompt engineering 最大的差別。它不是研究怎麼把一句提示詞寫得更漂亮，而是研究資訊流。哪些資訊該先出現，哪些應該被檢索後才出現，哪些根本不該在這一步干擾模型。

## 真正的敵人不是容量不夠，是髒亂

文章裡提到幾個很準的失敗模式。

第一種是 context poisoning。錯誤資訊一旦被塞進上下文，後面每一步都可能沿著同一個錯誤繼續長。模型不會主動知道它吃到的是毒，只會很認真地把毒消化成看似合理的回答。

第二種是 context distraction。歷史太多、背景太雜，模型開始被舊訊息牽著走，明明當前任務已經變了，卻還在延續前面的方向。

第三種是 context confusion。工具很多、文件很多，結果每個都沾一點，最後該用哪個工具、該看哪份資料反而更模糊。

第四種是 context clash。不同來源互相矛盾，但系統沒有先整理，模型只好在衝突裡硬選一個版本繼續往下跑。

你會發現，這些問題都不是單純把 context window 做大就能解。桌子變大，不代表房間就整齊。

## 六大支柱這種說法，真正有價值的是把責任拆開

這篇文章把 context engineering 拆成 agents、query augmentation、retrieval、prompting、memory、tools 六個部分。這種分法最大的好處，不在於分類本身，而是它提醒我們，上下文不是單一物件，而是一條供應鏈。

使用者的原始問題，通常根本不適合直接餵給每個元件。檢索系統需要的是比較適合查找的 query，LLM 需要的是整理過、可用的材料，工具需要的是格式正確的參數。把這些都叫做 prompt，反而會把問題看扁。

特別是 memory 這一層，很多產品最容易偷懶。短期記憶、長期記憶、工作記憶其實是不同東西。最近幾輪對話該留多少，哪些偏好值得長存，哪些暫時推導結果只在這次任務裡有用，這些不先分，系統之後就會一團亂。

## 記憶最難的不是存下來，是決定要忘什麼

我一直覺得，agent memory 真正困難的地方不是 persistence，而是 selection。

今天技術上要把對話、偏好、檢索片段、工具輸出存進資料庫，已經不算太難。難的是，什麼值得在未來重新浮現。你若把低品質、過時、含糊的資訊也一起留下來，下次檢索命中它們時，系統就等於替自己埋雷。

所以記憶系統做得好不好，不是看你留了多少，而是看你捨不捨得刪。這件事很反直覺，因為工程師很容易把「保留全部」誤認成保險。對 agent 來說，太多沒有被整理過的記憶，常常不是資產，而是負債。

## 工具使用其實也是上下文問題

很多人談 agent tool use，會把重點放在 function calling 格式、參數 schema、MCP 標準化。這些都重要，但更深一層的問題仍然是 context。

Agent 只有在知道有哪些工具、什麼情境該叫哪個工具、工具回來的結果要怎麼再送回推理鏈時，工具才真的存在。否則它只是掛在牆上的一排 API 名字。

也因此，工具數量不是越多越好。沒有整理過的工具清單，本身就是一種干擾。你替 agent 新增能力的同時，也在替它增加選錯的空間。好的 context engineering，連工具都會被分層、被裁切、被按任務召喚，而不是一股腦攤開。

## 我怎麼看這波 context engineering 熱潮

我同意這個方向會越來越重要，尤其在 coding agent 和多步驟工作流裡。因為這些場景對上下文錯誤的容忍度更低。聊天答錯一句，也許只是尷尬。程式改錯一段、工具叫錯一次，後果就可能直接進 repo。

但我也覺得，現在很多文章講得太像新名詞革命。其實 context engineering 真正說穿了，就是資訊架構。只是以前我們替人設計導航、替後端設計資料流，現在換成替模型設計工作桌。

如果你把這件事想清楚，很多 agent 問題都會變得比較具體。它不是神祕失憶，也不是模型忽然叛逆，而是該進來的資訊沒進來，不該留下的東西又一直擺在桌上。

Agent 不會自己把房間整理好。這件事還是工程師得做。
