## 真正卡住這個字典的，不是查詢速度，而是它背後那一大坨重複字尾

Andrew Quinn 這週寫了一篇很工程師的文章：他替自己的芬蘭語字典 `Taskusanakirja` 做增量搜尋，原本用 trie 還撐得住，等到資料從大約 40 萬個詞條膨脹到 4000 萬到 6000 萬個屈折變化，整個做法就開始失去比例感。單純的前綴查詢其實不慢，麻煩在於你得把那些變化形都帶在身上，而芬蘭語偏偏又是高度黏著語，一個詞根後面可以接出一長串反覆出現的尾巴。於是他先退一步，把這批資料塞進 SQLite FTS，功能是救回來了，代價是使用者第一次下載就得吞下一個 3 GB 的資料庫。

3 GB 會讓人停下來，不是 SQLite 慢，而是它太通用。SQLite FTS5 的設計目標，是把大量文件拆成 token，建立可查詢的全文索引，還能支援 prefix、phrase、NEAR、rank 這些查詢模式。它擅長的是「我有很多文件，想找含有某些詞的那幾篇」，不是「我有一大堆高度重複的字串尾巴，想把它們壓到極小，還保留前綴搜尋」。問題一旦收斂成字典查詢，通用索引仍然能做，只是你會為多餘能力一起付租。

## 舊方法不是不夠聰明，而是它只會共享前面，沒辦法共享後面

文章最有意思的地方，在於它把 trie 跟 FST 的差別講成了一個很具體的空間問題。trie 對前綴搜尋很自然，因為只要共享共同開頭就夠了；`kadun` 跟 `kaduille` 會一起走過前面那幾步。可是在芬蘭語這種語料裡，重複不只長在前面，更多時候是長在後面。大量詞形可能都收在相似的格位、複數、所有格與語助詞結尾上，這些尾巴在 trie 裡得一條一條再長一次。

BurntSushi 那篇經典長文講得更完整：有限狀態機拿來裝資料時，關鍵不是它會不會跑，而是它能不能把結構相同的路徑合併掉。`fst` crate 的文件也直接寫明，它用有限狀態機表示有序的 set 或 map，會同時壓縮共同前綴與共同後綴，還能在壓縮狀態下直接查詢。這正好打中 Andrew 的資料形狀。原本那個 3 GB SQLite 資料庫，換成 Rust 小工具抽資料、重建成 FST 後，只剩 10 MB 左右，縮了大約 300 倍。

這裡真正讓人記住的，不是「Rust 很省」，也不是「換資料結構就飛天」。更精確的說法是：當資料天然長得像一堆共享尾巴的有序字串，FST 會把那些重複路徑折起來；而 SQLite FTS 不會主動替你做這件事，因為那不是它被設計來優先處理的目標。

## FST 的甜蜜點很窄，但一踩中就會像作弊

如果你只看壓縮比，很容易把 FST 想成搜尋領域的萬靈丹。其實 `fst` 官方文件反而把限制寫得很老實。它適合存有序的 byte-string set 或 map，map 的 value 還限制在 `u64`，建表時必須按字典序插入；若資料沒排好，前處理就得自己做。另一個現實代價是查詢常常近似隨機存取，因此磁碟與 page cache 的行為會直接影響延遲。

FST 的強，不在「功能比較多」，而在「需求已經窄到足以讓它專攻」。Andrew 這個字典剛好全中：資料在執行期幾乎是靜態的，查詢型態高度集中在前綴搜尋，字串之間又存在大量共同後綴。`fst` 還支援 memory map，於是整個索引可以維持成一個很薄的二進位檔，交給作業系統決定哪些區塊真的要進記憶體。對想把產品塞進老筆電、單檔安裝包，或離線工具的人來說，這種差距不只是漂亮數字，而是你能不能交付。

反過來看，如果你的資料會頻繁更新、查詢不只 prefix、還想混進複雜排序、交易一致性、關聯欄位或多種條件過濾，那就別硬把所有問題都壓進 FST。SQLite FTS5 之所以胖，是因為它本來就在扛一整個比較泛用的搜尋系統。

## 真正值得記住的，不是 FST，而是先把問題做對，再把資料結構換對

Hacker News 上有一個回應我很認同：有時候最有價值的不是終極解，而是一個能讓你先把產品跑起來、再看清楚瓶頸長在哪裡的笨方法。Andrew 自己也把這件事寫得很坦率。九個月前，他先用 SQLite FTS 把功能救起來；九個月後，他已經更清楚自己手上的不是通用全文搜尋，而是一個帶著巨大形態學規律的字典索引。這個理解比「重寫成 Rust」更重要。

我看這篇的收穫，是重新提醒自己：很多看起來像效能優化的勝利，真正贏的其實是問題建模。先用通用工具把產品做活，等你知道哪些重複是假負擔、哪些結構才是真瓶頸，再回頭把資料交給更窄、更狠的結構，才會出現這種級別的縮減。FST 不是每個系統都該帶的一把刀，但碰到對的資料形狀，它確實會打得很準。
