每隔一陣子，就會有人宣告 MVVM 過時了。

理由通常也差不多。現在都 declarative UI 了，框架本身就很會管狀態，還要什麼 ViewModel。或者反過來說，大型系統都走 Clean Architecture、MVI、Redux 這些更完整的套路了，MVVM 聽起來像上一代的詞。

這種說法有一半對，另一半不對。對的地方是，MVVM 確實不再像十年前那樣，以一套明確模板出現在每個簡報裡。不對的地方是，它根本沒有消失，它只是被揉進今天的 UI 開發習慣裡，換了名字、換了語法，繼續活著。

## 為什麼它一直死不了

原因其實很樸素，UI 混亂這個問題從來沒被解決。

只要你在做互動式介面，就會一直遇到同樣幾件事。資料從哪裡來，畫面怎麼更新，按下按鈕之後誰負責處理，哪些邏輯該留在畫面，哪些不該。框架可以幫你把寫法變得漂亮，卻不會替你決定責任邊界。

MVVM 的核心價值，正好就是在這裡。View 只顧呈現，Model 管資料與商業邏輯，ViewModel 承接畫面狀態與操作流程。這個分法聽起來老派，但它之所以一直有效，是因為它對應的不是某個時代的 API，而是 UI 專案反覆發生的混亂。

換句話說，只要你的畫面不是靜態海報，MVVM 背後那個問題意識就還在。

## 現代框架看起來不同，本質卻很熟悉

WPF 時代的 MVVM 很明顯，你會看到 `INotifyPropertyChanged`、Command、Binding，一整套工程感十足的儀式。到了 SwiftUI，很多人以為自己早就不寫 MVVM 了，可你回頭看 `ObservableObject`、`@Published`、`@StateObject`，那不過是把同一件事做得更順手。

Flutter 也是一樣。它沒有硬性規定 MVVM，但只要你開始把畫面狀態抽到 `ChangeNotifier`、Riverpod 或其他 state holder，並讓 View 只負責 render，本質上你還是在處理 View 與邏輯分離。Blazor、Jetpack Compose 也沒有真正跳出這個問題域。

所以我一直覺得，與其問「MVVM 還在不在」，不如問「你有沒有把 UI 邏輯和畫面拆乾淨」。若有，你多半已經走在 MVVM 旁邊了，只是沒把那塊牌子掛出來。

## 真的有必要嗎，要看專案痛點

MVVM 最大的優點，從來不是它很優雅，而是它在專案開始變大時，能延後混亂爆炸的時間。

一個只有兩三頁的小 App，其實很常不需要這麼完整的分層。你硬要為每個畫面配一個 ViewModel，外加 repository、use case、binding abstraction，最後最重的不是功能，而是你自己的結構。

但一旦需求開始多，畫面狀態開始複雜，非同步請求、表單驗證、載入與錯誤處理糾纏在一起時，MVVM 的價值就會浮出來。它幫你把原本會長在 View 裡的東西搬出去，至少讓測試與除錯有一個落腳點。

我很常把它想成一種節制。不是所有專案都需要，但當你發現 View 已經開始知道太多事，MVVM 往往是最先派得上用場的那把工具。

## 它真正的弱點，也不是大家嘴上的老問題

很多文章講 MVVM 缺點，第一句就是 boilerplate 太多。這在今天其實沒那麼致命，因為不少框架和工具已經幫你把樣板折掉了。

我覺得真正麻煩的是另一件事，ViewModel 很容易長胖。

當團隊把「不該在 View 裡的東西」全部塞進 ViewModel，它就會從隔離層慢慢變成垃圾桶。API 呼叫、驗證規則、導航狀態、暫存資料、畫面轉換、錯誤文案，最後全部擠在同一層。你表面上避開了 Massive ViewController，實際上只是長出 Massive ViewModel。

所以 MVVM 從來不是分完三層就結束，它後面還需要更細的節制。哪些邏輯應該再下沉到 use case，哪些狀態應該被模組化，哪些 shared state 不該偷偷長成全域依賴。真正難的是這些，而不是把名字取對。

## 它和 Clean Architecture 不是競爭，是前後關係

2026 年再看，MVVM 最合理的位置，其實已經不是「整個系統的答案」，而是 presentation 層的一種做法。

Clean Architecture 負責更大尺度的邊界。MVVM 則處理畫面與互動這一層怎麼保持乾淨。

如果把 MVVM 當成萬用架構，當然會失望。它無法獨自回答所有問題。但若把它放回 UI 層，它又非常耐用。這也是它一直沒有退場的原因，它不是皇帝，卻是一個很好用的中層管理者。

## 我對 MVVM 的實際判斷

如果你今天要做一個中型以上的前端或 App 專案，我仍然會把 MVVM 放進選項，而且通常不會排太後面。不是因為它最潮，而是因為它夠穩，團隊也容易理解。特別是在需要測試、需要多人協作、需要持續演化畫面的情況下，它提供了一個足夠清楚的共同語言。

但我也不會把它當信仰。小專案別硬上，狀態極度複雜又要求單向資料流可追蹤性的系統，也許 MVI 更合適。架構模式不是徽章，不是你戴上就代表成熟。

MVVM 到今天還重要，不是因為它從未改變，而是因為軟體介面始終會失控，而它剛好一直都能在失控前，替你擋住一部分混亂。這種能力，看起來不新，卻很難被淘汰。
