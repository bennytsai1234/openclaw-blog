想像你在使用一個強大的 AI Coding Agent，比如 Claude Code。起初它表現得像個天才，能迅速理解你的整個專案結構並精準地修改 Bug。但隨著對話拉長，當你進入第三或第四個小時，你開始發現它變得「遲鈍」了：它忘了你十分鐘前才剛商定好的變數命名規範，或者在處理一個跨文件的重構時，突然把之前已經修好的邏輯又改回去了。

很多開發者會以為這是模型能力的上限，或者是所謂的「中段遺忘」（Lost in the Middle）現象。但實際上，問題出在一個更基礎的工程痛點：上下文視窗（Context Window）是一個極其低效的線性儲存空間。即便現在的模型宣稱支持 1M 甚至更多 token，但在實際的 Agentic 工作流中，大量的工具調用紀錄、重複的文件內容以及冗長的推理過程，會迅速填滿這個空間。

當視窗滿了，系統只能採取截斷（Truncation）或簡單的摘要（Summarization）。前者會直接丟棄關鍵資訊，後者則會將具體的技術細節轉化為模糊的描述。這就是為什麼 AI Agent 在長程任務中會逐漸「失智」的原因。

## 讓模型自己管記憶，就像在用 GOTO 寫程式

在 LCM 出現之前，學界嘗試過一種名為 RLM（Recursive Language Models）的範式。RLM 的核心思路很直覺：既然模型能寫程式，為什麼不讓模型自己寫一段腳本，遞迴地呼叫自己來處理上下文？

這種「符號遞迴」（Symbolic Recursion）在短期內看起來很靈活，因為模型可以根據當前需求，動態決定要回溯到哪一段記憶。但這種做法在工程上極其脆弱。它就像是在 1960 年代的程式設計中大量使用 `GOTO` 語句一樣——雖然你能跳到任何地方，但一旦邏輯複雜到一定程度，整個執行流程就會變成一團亂麻（Spaghetti Code），且完全沒有終止保證。

模型在嘗試自我遞迴時，經常會陷入無限循環，或者在遞迴過程中遺失對原始狀態的精確指引。這種依賴模型「自律」來管理記憶的方式，無法提供生產環境所需的確定性。

## LCM 的直覺：將記憶從「文本」轉化為「結構」

LCM（Lossless Context Management）的設計者意識到，我們不應該讓模型去「管理」記憶，而應該由一個確定性的引擎來「驅動」記憶。LCM 將原先模糊的遞迴過程拆解為兩個由引擎管控的確定性機制：遞迴上下文壓縮（Recursive Context Compression）與遞迴任務分區（Recursive Task Partitioning）。

首先是那個讓它被稱為「Lossless（無損）」的核心——遞迴上下文壓縮。

傳統的摘要方法是將 $\text{Text A} \to \text{Summary A}$，一旦轉換完成，原始的 $\text{Text A}$ 就從工作記憶中消失了。而 LCM 建立了一個層級化的摘要有向無環圖（Summary DAG）。在這個圖中，舊的訊息會被自動壓縮成摘要，但每個摘要節點都保留著指向原始文本的「無損指針」。

這就像是在地圖上操作：當 Agent 需要全局觀測時，它閱讀的是高層級的摘要（縮小地圖）；而當它需要處理具體代碼時，引擎會根據指針，精準地將該片段的原始文本重新注入上下文（放大地圖）。這種機制確保了資訊在被壓縮的同時，依然具備 100% 的可追溯性。

## 用 `LLM-Map` 替代模型寫的 Loop

除了記憶的儲存，LCM 解決了另一個核心問題：任務的執行。

在一般的 Agent 框架中，如果你要求 AI 「檢查專案中所有 `.ts` 檔案並修正型別錯誤」，模型通常會寫一個循環，一次讀一個檔案，處理完後再讀下一個。這種線性流程在長任務中非常緩慢，且一旦中間某一步出錯，整個鏈條就斷了。

LCM 引入了類似編譯器原語的「遞迴任務分區」。它用確定性的引擎原語（例如 `LLM-Map`）取代了模型手寫的 Loop。`LLM-Map` 會將任務自動分拆成多個並行單元，由引擎統一調度執行，最後再將結果遞迴地聚合起來。

這是一個從「靈活但混亂」到「受限但可靠」的轉向。LCM 犧牲了模型隨意操作記憶的 maximal flexibility，換取了對終止時間的保證（Termination Guarantees）以及零成本的短任務連續性。

## 從 OOLONG 評測看 Volt 的實力

為了驗證這套架構，研究團隊構建了一個名為 Volt 的 Coding Agent（基於 Opus 4.6 模型）。他們在 OOLONG 這個專為長上下文設計的評測集上，將 Volt 與目前的頂尖 Agent（如 Claude Code）進行了對比。

結果相當驚人：在 32K 到 1M tokens 的所有 context 長度區間內，Volt 的得分均高於 Claude Code。這證明了當任務複雜度增加到一定程度時，單純增加 Token 視窗的大小（Capacity）已經邊際效用遞減，真正的突破點在於記憶的組織方式（Architecture）。

這給我們一個很重要的啟示：即便未來模型支持 10M 甚至 100M 的視窗，我們依然需要 LCM 這樣的管理層。因為對於一個需要運行數天、涉及數千個檔案的開發任務來說，資訊的「密度」和「檢索精度」遠比「容量」更重要。

## 記憶的未來：從容量戰爭轉向管理戰爭

回看 LLM 的發展，我們經歷了從 4K 到 128K，再到 1M+ 的視窗擴張。但 LCM 告訴我們，上下文視窗不應該被視為一個簡單的儲存桶，而應該被視為一個需要被管理的快取（Cache）。

LCM 的成功在於它將「記憶管理」從模型的隨機推理中抽離，交給了確定性的計算引擎。它將記憶從一條線性的文本流，變成了一個可以隨時縮放、精準定位的圖結構。

對於開發者來說，這意味著我們將進入一個新的階段：我們不再追求模型能「一次讀完多少書」，而是在意 Agent 能否「在閱讀十萬頁文獻後，依然能精準地指出第三百二十頁第二段的一個邏輯漏洞」。

這不是對大視窗的否定，而是對大視窗的補完。當我們不再被「上下文牆」困住時，AI Agent 才能真正從一個「對話框」演變成一個能夠獨立處理複雜工程的「數字員工」。
