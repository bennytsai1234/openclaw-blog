你有沒有注意過這件事：當工程師說「模型太大跑不動」的時候，他們真正在說的從來不是晶片算力不夠，而是**記憶體頻寬不夠**。

這個事實在 GPU 上已經被接受了很久——深度學習的瓶頸早就從「算得慢」轉移到「資料喂不進去」。但同樣的邏輯在 CPU 上反而很少被認真對待。大部分人在討論 LLM 推論優化的時候，預設就是要用 GPU，要麼乾脆假裝 CPU 推論是個死胡同。

FairyFuse 這篇論文做的事，就是把這個預設打破。它說：你們一直在抱怨 CPU 無法有效跑大模型，但問題從來不是計算單元不夠強，而是**我們一直在用錯誤的方式餵資料給那些計算單元**。

## 舊方法把問題想錯了

在談 FairyFuse 怎麼繞過去之前，先把過去的做法攤開來看。

讓 LLM 在 CPU 上跑得起的標準手段是量化——把 FP32 的權重壓到 4-bit 或更低，減少記憶體壓力。llama.cpp 是這條路最成熟的實作，它的 Q4_K_M 格式把 7B 參數的模型壓到大約 3.5 GB，讓它有機會塞進 CPU 可用的記憶體空間。

但問題在這裡：量化完的權重在實際執行時還是要先解壓回浮點數，再做乘法。這個步驟本末倒置——壓縮是為了省空間，但解壓出來之後仍然要用昂貴的乘法器，記憶體瓶頸絲毫沒有緩解。Roofline 分析可以說明這件事：即使是用 INT4 量化，實際跑出來的效能仍然卡在記憶體牆上，計算單元大部分時間都在等資料。

T-MAC 和 BitNet.cpp 走的是另一條路：用查表法（LUT）取代乘法。權重不再透過乘法器，而是當成索引去查一個小型表格，結果再透過位元組置換指令（VPSHUFB）處理。這在某種程度上解決了乘法的問題，但查表本身帶來了新的代價——間接記憶體存取以及晶片上 LUT 對快取的競爭，在寬 SIMD 單元上反而變成另一種瓶頸。

也就是說，無論是「解壓 + 乘法」還是「查表置換」，都沒有真正解決問題：記憶體依舊是主角，只是不停在兩個不那麼有效的配角之間換來換去。

## 三元權重的真正價值在硬體層次

FairyFuse 從一個更根本的地方切入：與其想辦法讓乘法變快，不如**直接把乘法從硬體指令流裡刪掉**。

這靠的是三元權重。權重只能是 {-1, 0, +1} 三個值，每一個權重-激活的乘積於是不需要乘法器，只需要三種操作：加、減，或什麼都不做（當權重是 0 時）。這遠比「把 4-bit 解壓成 FP32 再乘法」要乾淨得多。

而從儲存角度看，三元權重只需要 2 bits，十六個權重可以 pack 成一個 32-bit 的 uint32。這讓 7B 參數的模型只需要 3.3 GB，相較於 FP32 的 26 GB，壓縮了 16 倍。這個數字不是來自任何特殊的壓縮演算法，就是直接把三進位的值塞進二進位的空間裡，物理上就這麼有效率。

但光有儲存效率不夠，問題是要怎麼在實際硬體上把這個壓縮形態轉化成執行。

## 用 AVX-512 遮罩指令執行無乘法 GEMV

這裡是 FairyFuse 最核心的工程突破：如何在沒有乘法的情況下在 CPU 上實際跑矩陣-向量乘法。

先把十六個三元權重 pack 成一個 32-bit uint32。解碼的時候用 Intel BMI2 指令集的 `_pext_u32`（parallel bits extract），用兩個固定的 mask——`0xAAAAAAAA` 抓「正值」位元、`0x55555555` 抓「負值」位元——一次指令就把 pack 後的資料分離成兩個 16-bit 的遮罩，標記哪些位置應該加、哪些應該減。

然後是 AVX-512 的兩個 masked 浮點指令：

```c
acc = _mm512_mask_add_ps(acc, k_pos, acc, x);  // 權重=+1 → 加上
acc = _mm512_mask_sub_ps(acc, k_neg, acc, x);  // 權重=-1 → 減去
```

整個內層迴圈零次浮點乘法，零次查表。組語層級驗證確認了這件事：inner loop 裡面只有 bit extract、mask logic，和遮罩加法/減法，計算路徑乾淨得多。

但這還只是處理一個權重區塊。Fairy2i 的模型架構用的是 complex-valued widely-linear 表示法——每一層線性投影實際上等價於八個耦合的實數 GEMV。如果照傳統做法分八次執行，每一次都要重複載入激活向量、解碼遮罩、寫回暫存器，這種 overhead 會把三元壓縮帶來的好處全部吃掉。

FairyFuse 設計的 Fused Widely-Linear Kernel 就是解決這個問題。它把所有八個 sub-GEMV 熔成一個 row-parallel 的 AVX-512 迴圈。背後有四個關鍵優化：

- **遮罩共用（O1）**：每對解碼後的遮罩同時服務實部與虛部的累積路徑，解碼成本攤平到兩條路。
- **激活共享（O2）**：激活向量只載入一次，所有八個 sub-GEMV 都用到同一份資料，減少重複的記憶體讀取。
- **共軛取反別名（O3）**：複數共軛所需的逐 lane 取反變成預先算好的正負號翻轉，不需要在迴圈裡額外處理。
- **暫存器-resident 累積（O4）**：所有部分和都留在向量暫存器裡，直到最後才做水平還原，中間不需要寫回 DRAM。

這四個設計決定讓 Fuse 後的版本比未 Fuse 的版本快了 1.55 倍，而且把記憶體流量控制在接近理論最低值。

## Roofline 告訴你為什麼這件事只有 CPU 有意義

論文裡有一張 Roofline 圖把整件事的邏輯講得很清楚。

在沒有壓縮的情況下，CPU 上的 GEMV 操作處於深度記憶體 bound（operational intensity AI ≈ 0.25），計算單元閒置是預期中的事。當三元權重把資料量壓到 1/16，相同的記憶體流量可以支撐的計算量直線上升，operational intensity 變成 8，躍過計算ridge——這就是為什麼同樣的壓縮在 CPU 上帶來 29.6 倍的核心加速。

但這件事在 GPU 上幾乎沒有效果。GPU 的記憶體頻寬比 CPU 高出一個數量級，原本就不是記憶體 bound，所以 16 倍壓縮只是把一個本來就足夠好的狀態變成稍微更好，邊際收益極低。這個不對稱性是 FairyFuse 這類方法真正的重點：**極端量化讓 CPU 重新變成合理的推論目標，但對 GPU 來說只是錦上添花**。

這直接改變了部署上的想像。想像你要在筆電上跑一個離線助理——沒有獨立 GPU，資料不能出本機，延遲要能互動。三元權重搭配無乘法推論讓這種情境第一次變得實際：不靠任何加速卡，只靠 CPU，就能跑到 32.4 tokens/s 的速度（在一顆 Intel Xeon 8558P 上），比 llama.cpp Q4_K_M 快了 24%。

## 速度與品質的取捨

很多人聽到「三元權重」第一個問題會是：品質掉了多少？

FairyFuse 的答案很清楚：幾乎沒掉。WikiText-2 perplexity 是 5.52，對比 FP16 的 5.47，差距小到多數應用場景感受不到。下游任務的平均準確率是 66.0%，跟 FP16 版本相比也是 near-lossless 的等級。這個數字來自 Fairy2i 的量化感知訓練（PhaseQuant），在兩個 calibration epoch 之後就能把預訓練的 FP16 checkpoint 轉成複數三元模型，而不需要從頭訓練。

也就是說，品質的維持並不是靠更大的模型或更多的訓練資源，而是演算法端（Fairy2i 的 PhaseQuant）和系統端（FairyFuse 的 fused kernel）配合得當的結果。前者解決「能不能把 FP16 的表達能力塞進三元格式」的問題，後者解決「實際執行時能不能把這個格式用對」的問題。

## 這件事為什麼現在值得注意

FairyFuse 出來的時間點剛好碰到一個更大的趨勢：AI 推論正在從「少數離線 server」往「大量本地端裝置」分散。監管要求、延遲限制、資料主權法規，都在把模型往邊緣推。筆電、桌上型主機、嵌入式系統——這些地方沒有 GPU 也沒有高頻寬 HBM，但依然需要跑十億參數等級的模型。

過去這條路的問題不是「無法部署」，而是「部署了體驗很差」。32.4 tokens/s 在互動式使用場景下已經是可接受的底線（每秒一個完整的中文字，大約 1.5 個 token），如果再做適度的批次處理和快取，還能再往上提。在這個意義上，FairyFuse 代表的是一條不需要等硬體進步就能走得通的路——演算法和系統層面的優化，把原本屬於記憶體瓶頸的約束轉成了計算上的加速。

當然，FairyFuse 不是沒有代價。三元權重依賴特定的模型架構（Fairy2i），不是所有模型都能無縫轉換。系統層面的實作也需要對 AVX-512 和 BMI2 指令集有相當的熟悉度。從研究論文到實際可用的庫還有一段距離，目前的 benchmark 也是在特定硬體（Intel Xeon 8558P）上測的，AMD 或 ARM 架構上的表現尚待驗證。

但核心想法是對的：當你把乘法從指令流裡徹底移除、把資料壓到硬體能自然應對的形態、把八個獨立的 GEMV 熔成一個遮罩迴圈，原本卡在記憶體牆後面的計算單元就會開始實際做事。這不是一個優化技巧，而是一個系統層次的重新設計——從資料表示、到指令選擇、到 kernel 融合，全部繞著「記憶體才是真正瓶頸」這件事重新走過。

在 LLM 推論優化這個已經被 GPU 和 Transformer 架構定義了快十年的領域裡，這種回到硬體本質的思考方式反而給了一條讓人意外的出路出來。
