有些工程假設，大家用久了，就不太再檢查了。

KV cache 就是其中一個例子。這幾年它幾乎成了所有自回歸推理系統的基本配備，因為不快取幾乎沒法做實用部署。久而久之，很多人自然把另一件事也一起默認了，只要模型、輸入、硬體都一樣，cache-ON 和 cache-OFF 應該只差在速度，不差在答案。

這篇新論文偏偏就是拿這個默認開刀，而且結果還很難忽視。在標準 FP16 下，兩條執行路徑會穩定分岔，不是偶發，不是抽樣波動，而是每次都會偏。換成更直白的說法，快取一打開，你跑的就已經不是完全同一條數值軌跡了。

## 問題不在隨機性，而在浮點的秩序感

作者把實驗設計得很乾淨。他們在 LLaMA-2-7B、Mistral-7B-v0.3、Gemma-2-2B 上，比較 cache-ON 和 cache-OFF 的輸出。為了排除抽樣因素，連 greedy decoding 都測了。

結果很狠，所有模型、所有策略下，都觀察到 100% 的 token 偏移。也就是說，兩條路最後一定不同。

這裡最關鍵的點，是原因並不神祕。FP16 本來就不是結合律成立的數值世界。你把一串加總用不同順序做，最後就可能得到不同結果。cache-ON 和 cache-OFF 恰好不只是在做「同樣計算的省略版」，而是走了不同的 kernel 佈局、不同的記憶體排布、不同的累積順序。於是誤差不是噪聲，而是制度性的。

論文也做了最重要的對照，一旦換成 FP32，偏移幾乎消失。這基本上把兇手釘死了，不是模型性格，不是資料集特性，就是 FP16 的累加路徑差異。

## 為什麼 GQA 會讓事情更糟

我覺得這篇很有價值的一點，是它沒有停在「有偏差」這個結論，而是往下追偏差怎麼擴散。

Mistral 用的是 GQA，多個 query heads 共用同一組 KV heads。這本來是很實用的效率設計，但副作用是，一個數值誤差不再只影響單一路徑，而會被一起廣播給多個 heads。錯誤不是散開，而是同步擴大。

這讓問題變得很有意思。原本我們常把注意力架構的選擇，理解成速度、記憶體和品質的取捨。現在多了一層，還有數值穩定性。某些架構不只是更省，也可能更會把小誤差變成集體偏移。

## 殘差流補不回來，代表真正變的是狀態

論文還做了 activation patching，嘗試把殘差流補回 cache-OFF 的軌跡。結果沒有救回來。這個實驗很重要，因為它告訴你，偏差不只是中間表示某幾層歪掉，而是 KV cache 這個狀態本身已經走上另一條路。

一旦狀態不同，後面所有 token 都是在新的地板上繼續蓋。你不太可能靠補一補 residual stream，就把整座樓拉回原來的位置。

## 這件事對工程有多實際

我覺得很實際，尤其對那些重視可重現性的人。

很多團隊會在離線驗證、學術實驗、甚至 debug 時使用較高精度或不同推理設定，然後把 production 交給標準 FP16 cache 路徑。以前這樣做的時候，大家多半只擔心速度和成本，不太擔心語義上會不會分岔。這篇論文等於提醒你，會，而且是可預期地會。

更微妙的是，cache-ON 在多數條件下甚至比較準。這說明偏差不是單純退化，而像是某種穩定但不同的數值偏置。它不保證更差，也不保證更好，只保證不一樣。

## 我怎麼看這個發現

我喜歡這篇，不是因為它推翻了 KV cache，而是因為它把一個大家太習慣的工程近似重新照亮。很多基礎設施之所以危險，不是因為它錯，而是因為它太早被當成理所當然。

對大多數系統來說，這個偏移未必會立刻變成災難。可只要你的場景在乎確定性、在乎嚴格重現、在乎線上線下結果對齊，這件事就不能再被當成小數點誤差。

我們過去把 KV cache 視為純效率技巧，現在看來它其實也改變了計算本身。這不是語言遊戲，是部署現實。
