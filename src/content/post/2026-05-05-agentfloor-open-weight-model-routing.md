## 真正讓 agent 變貴的，常常不是推理本身

這兩天如果你有在看 agent 相關論文，最容易被記住的可能是另一篇談「tool-use tax」的研究：工具不是一定有幫助，格式、協定和多一步呼叫本身，反而可能先把模型拖慢、拖笨。這篇 **AgentFloor** 有意思的地方，在於它把同一個問題換了一個更接地氣的角度來問：既然 production agent 大多數請求，本來就不是那種史詩級長推理，而是搜尋、查表、抽欄位、送出一個 structured payload，那我們到底有多少工作，真的非得用 GPT-5 這種前沿模型不可？

這個問題比看起來更實際。今天很多團隊做 agent routing，常見做法還是「重要任務先丟大模型，剩下的再說」。原因也不難理解：沒有人想因為省了一點 token 或延遲，最後把整條 workflow 的穩定性賠進去。但 AgentFloor 想挑戰的，正是這種保守直覺。作者不先問哪個模型比較聰明，而是先把 agent 工作拆成不同階梯，然後量一件更工程化的事：**什麼複雜度以下，小型 open-weight 模型其實已經夠用了。**

## 他們不是做更大的 benchmark，而是先把雜訊拿掉

Agent 評測最麻煩的地方，通常不是分數低，而是你不知道那個分數到底在量什麼。像 ToolBench、WebArena、OSWorld 這類 benchmark 很貼近真實環境，但也因此混進太多外部變因：API 會漂移、網站會改版、GUI 狀態不穩、甚至還有資料污染的疑慮。最後你看到某個模型失敗，很難判斷它是真的不會規劃，還是只是剛好卡在環境噪音。

AgentFloor 的設計思路剛好反過來。它只做 **30 個 deterministic tasks**，但把它們排成六層 capability ladder：從 **A0 的純指令遵循**、**A 的單次工具呼叫**、**B 的兩步串接**、**C 的條件分支**、**D 的多來源綜合與衝突修復**，一路到 **E 的長鏈規劃與持續約束追蹤**。工具也固定成 8 個抽象操作，全部跑在記憶體中的 fixture database，沒有真實 API、沒有檔案系統、沒有時間變動狀態。

這樣做的代價，是它不像開放世界 benchmark 那麼熱鬧；但好處也很明顯：你終於可以比較乾淨地看見，失敗到底是模型能力問題，還是環境偶然性。對工程團隊來說，這種設計反而更有參考價值，因為 production routing 最怕的不是「模型在網頁上按錯按鈕」，而是你連哪一類任務本來就不該花大錢都搞不清楚。

論文共跑了 **16 個 open-weight 模型**，尺寸從 **0.27B 到 32B**，再加上 **GPT-5**，總計 **16,542 次 scored runs**。這種規模不算天文數字，但已經足夠支撐它的核心主張：在 agent pipeline 裡，真正需要前沿模型的部分，比大家想像得窄。

## 最有意思的結果，不是誰贏，而是分界線真的長出來了

AgentFloor 最值得看的不是 overall leaderboard，而是每一層能力階梯的切面。論文裡拿來當 open-weight 代表的是 **gemma4:26b**。整體來看，它和 GPT-5 在 270 個 paired observations 上幾乎打平，差值只有 **+0.4 個百分點**，而且在作者預先註冊的 **±10 個百分點 equivalence margin** 內成立。換句話說，如果你只看 aggregate score，結論其實已經很刺激：至少在這套受控 benchmark 上，最強的 open-weight 模型已經能和 GPT-5 打到統計上等價。

但更重要的是 tier breakdown。**A0 層**，也就是完全不用工具、只看指令遵循時，gemma4:26b 反而是 **100% 對 80%** 贏 GPT-5。**A 層** 的單工具呼叫，兩者也正式達到等價。到了 **B、C、D** 這三層，也就是 sequential chaining、branching 與 multi-source synthesis，open-weight 模型雖然沒有拿到足夠窄的信賴區間去宣告等價，但 point estimate 已經非常接近 GPT-5。

真正拉開差距的是 **E 層**。這一層要求模型在多步流程中持續追蹤 constraint、維持規劃狀態，GPT-5 的通過率是 **10.2%**，gemma4:26b 則是 **0%**。表面上看起來像碾壓，但其實這個數字也暴露另一件事：**前沿模型確實比較強，可是它也遠遠沒有強到能放心上 production。**

這就是這篇論文最誠實的地方。它沒有把結論寫成「小模型全面追平大模型」，也沒有反過來說「所以 routing 仍然應該以 frontier model 為主」。作者真正給出的判斷更像是一條 operational boundary：大量短鏈、結構化、例行性的 agent 呼叫，已經可以往小模型或中型 open-weight 模型下放；你該保留給前沿模型的，是那一小段需要長時間維持計畫、持續修正、不能丟失約束的工作。

## 省下來的不只是錢，而是整個系統設計的自由度

如果只把這篇論文看成 cost optimization，會低估它的價值。當然，成本數字很有衝擊力。論文用作者鎖定的 2026-04 價格計算，**gemma4:26b 在 Mac 上每個 passed task 約 0.0022 美元**，對上 **GPT-5 的 0.0327 美元**，大約便宜 **15 倍**；放到雲端 GPU，仍然有接近 **3 倍** 的差距。這些數字足以讓任何做 agent 產品的人停下來想一下 routing。

但更深一層的意義是：一旦你承認 agent pipeline 裡大多數步驟其實只是「短、窄、規格明確的工具工作」，整個系統設計就不需要再假設每一步都要綁定同一顆最強模型。你可以把查詢、抽取、欄位格式化、簡單判斷這些步驟切出去，交給便宜而穩的 open-weight 模型，把真正稀缺的前沿推理額度留給少數高風險節點。這不只是 token 成本問題，也是延遲、併發、可部署性，甚至是資料主權的問題。

這裡剛好能跟前面提到的「tool-use tax」互相對照。那篇論文說的是：工具不一定白拿，tool-calling protocol 本身就可能讓表現變差。AgentFloor 則補上另一半：就算工具值得用，也不是每一步都要配最昂貴的腦袋。兩篇合起來，其實在逼同一個產業習慣鬆動——**別再把 agent 想成單一模型不斷往前跑，而要把它當成一條需要分工、分層、分級配置的流程。**

## 真正還不能高估的，是長鏈規劃這件事

如果要說這篇論文哪裡最值得保留懷疑，我會放在它的外推範圍。AgentFloor 雖然刻意排除了 API 漂移和真實環境噪音，這讓能力邊界變得更清楚，但也表示它量到的是「在乾淨工具世界裡，模型能走多遠」。真實世界的 agent 系統通常還要面對 UI 不穩、資料缺漏、回傳格式不一致、權限問題、甚至使用者需求中途變動。這些因素很可能讓長鏈任務更難，也可能讓一些看似簡單的短鏈任務出現意外失敗。

不過這不會推翻論文的核心價值，反而讓它的結論更像一條安全邊界：**如果連在乾淨環境裡，長鏈規劃都只有 10.2% 對 0% 這種等級，那就更不該在真實 production 裡對 fully agentic long-horizon planning 抱太多幻想。** 反過來說，A 到 D 這些短中鏈工作既然已經接近，那些原本全丟給 frontier model 的 pipeline，就很值得重切一次。

我讀完最強烈的感覺是，AgentFloor 並沒有告訴我們「小模型已經贏了」，它告訴我們的是另一件更實用的事：**我們終於比較知道，該在哪裡停止迷信大模型。** 這對 2026 年的 agent 工程，可能比再多一份 leaderboard 都重要。
