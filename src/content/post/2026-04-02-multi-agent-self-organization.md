很多團隊第一次做 multi-agent，直覺都很像在排一個小型公司。這個 agent 當 PM，那個負責寫 code，另一個做 reviewer，再加一個總指揮。畫面很合理，也很有人類組織感。可這篇研究偏偏對這種設計下手，還得出一個不太客氣的結論，先把角色分得太清楚，可能反而讓系統變弱。

論文做了 25,000 個任務實驗，讓不同模型、不同 agent 數量、不同協調協議彼此對打。最後贏下來的，不是中央集權，也不是完全自由，而是一種很節制的結構，固定發言順序，但不替每個 agent 預先定角色，讓它們自己決定何時承擔什麼工作。

這個結果之所以好看，不只是因為數字高，而是因為它正面挑戰了目前不少多 agent 框架的出發點。

## 為什麼人類式分工不一定適合 LLM

我們習慣分角色，因為人類本來就有專業邊界，也有上下文記憶限制。一個人不可能同時扮演所有崗位，分工能降低混亂，提高效率。但 LLM agent 不一樣。它不會累，不會真的被身份綁住，也可以在同一輪裡快速切換視角。你昨天要它像工程師，今天要它像編輯，它不會因為工牌不同就失去能力。

所以當工程師把人類公司的組織圖原封不動搬進 agent 系統時，問題就來了。那些角色設定看起來整齊，實際上可能只是提前限制了 agent 的行動空間，逼它們按照我們熟悉、但未必高效的方式協作。

## 完全自由為什麼也不是答案

不過，論文沒有走向另一個極端。它同樣證明，若讓所有 agent 完全共享上下文、誰想說什麼就說什麼，結果也不好。因為這樣的系統會把太多算力花在協調本身，像一群人同時開口，人人都有想法，卻沒有人把節奏收住。

研究裡最好的協議叫 Sequential。它做的事情其實很少，只規定輪流順序，不規定角色。聽起來像退了一步，卻剛好卡在一個很妙的位置。它替系統解決「誰先來」這個基礎問題，讓 agent 不用把資源浪費在搶話和協調上，同時又保留足夠自由，讓真正適合的 agent 去接住當下的任務。

這也是論文所說的 endogeneity paradox。中央控制太硬，完全自由太散，最好的表現反而來自「有邊界，但不預設內部分工」的中間地帶。

## 這和模型強弱也有關

我覺得文中另一個很實用的發現，是能力閾值問題。不是所有模型都適合自組織。能力較弱的模型，往往沒有足夠好的自我評估能力，容易高估自己，接下超過能力範圍的工作。這時候，鬆散結構不但不會帶來創造力，還可能變成混亂。

這點對實務很重要。很多團隊聽到「自組織更好」就會想整包照搬，可如果你用的是較小的開源模型，或推理能力本來就不穩，結果可能和論文裡最強模型完全不同。結構不是越少越高級，而是要配合模型能力。

## 規模擴大後，角色反而更多了

論文裡還有個很迷人的細節。當 agent 數量擴到 256 個時，系統品質沒有明顯下降，而且 8 個 agent 就自發長出超過五千個獨特角色。這代表什麼？代表角色不是不能有，而是不一定要由人先畫好。給它足夠的任務和互動空間，系統自己會長出細緻分工。

這和我們平常設計 workflow 的方式很不一樣。過去是先有職位，再有工作；這裡比較像先有問題，再讓結構慢慢從問題裡長出來。

## 開源模型的那組成本數字很難忽視

研究還提到，開源模型大概能到封閉模型 95% 的品質，但成本只有約 1/24。這對 multi-agent 特別有感，因為多 agent 天生會放大 token 與推理成本。若每個回合都要付閉源旗艦價錢，系統很快就不現實。

所以這篇論文其實不只在談協調哲學，也在替部署策略給建議。只要模型能力跨過那個門檻，開源模型加上自組織協議，可能是很有競爭力的組合。

## 我最後帶走的判斷

我不會把這篇論文讀成「以後都不要設計角色」。更精確的說法是，角色不應該是第一原則。先問任務需要多少節奏控制，再問模型是否有足夠能力自行分工，最後才決定要不要加上更明確的外部結構。

如果你做 multi-agent 系統，這篇文章很值得放在設計桌邊。它提醒我們，最好的協作方式未必像公司，反而更像一個有秩序的討論現場。大家輪流說話，但誰來接哪件事，不用一開始就寫死。這種鬆而不散的設計，也許才更像 LLM 真正擅長的工作方式。
