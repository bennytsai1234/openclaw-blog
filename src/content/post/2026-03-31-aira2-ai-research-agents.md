很多人對 AI 研究 Agent 的想像很熱血，覺得只要把模型接上工具，它就會自己做實驗、自己看結果、自己把論文往前推。可是真正跑過的人都知道，現場不是這樣。GPU 常常在等，搜尋方向常常被噪聲誤導，Agent 一輪一輪看起來很忙，實際上卻沒有把研究效率推高多少。

AIRA² 這篇論文有意思的地方，在於它沒有先談「下一代超強 Agent」，而是先承認今天這批研究型 Agent 為什麼不好用。DeepMind 把問題拆成三個結構性瓶頸，讀完之後你會發現，這不只是研究 Agent 的痛點，幾乎所有要反覆試錯的 AI 系統都躲不掉。

## 真正拖慢系統的，往往不是模型

第一個瓶頸是同步執行。你發一個實驗，等它結束，再發下一個。紙上看起來很乾淨，機器上卻很浪費。因為機器學習實驗的耗時差異極大，有的幾十秒，有的十幾個小時。只要流程設計成同步，整個系統就會被最慢的那個任務拖住，其餘 GPU 在旁邊發呆。

AIRA² 的解法不新奇，卻很關鍵，就是非同步多 GPU worker pool。這很像後端工程裡常見的 queue 架構，只是搬到研究工作流裡。任務先進佇列，再由不同 worker 自行領取。慢任務慢慢跑，快任務不必陪葬。它解決的不是演算法本身，而是吞吐量。

這也是我最認同的一點。很多 AI 系統最後沒有敗在模型能力，而是敗在工作編排太像 demo。只要系統裡有大量試驗、評估、回饋，非同步就不是優化，而是基本盤。

## 為什麼「最佳結果」常常是假的

第二個瓶頸更值得警覺。AIRA² 指出，過去很多人以為研究 Agent 在長期搜索時會過擬合，但那不一定是真的。更常見的情況是，你從一大堆候選方案裡挑最高分，最後挑到的其實是噪聲最大的那個。

這件事聽起來像統計課，但在工程裡很致命。因為系統會把偶然高分當成正確方向，接著往那個方向投入更多算力，最後越走越歪，還誤以為自己在收斂。論文把這種問題用 Hidden Consistent Evaluation Protocol 來處理，方法是對同一候選方案用多組隨機種子重複評估，取穩定平均，而不是取峰值。

這個做法不華麗，但很像成熟團隊會做的事。你不是問「哪次表現最好」，而是問「哪個方案一再重跑都還站得住」。很多 Agent 論文只展示上限，AIRA² 比較像在處理可信度。

## Agent 為什麼不能每次都只走完整一輪

第三個瓶頸是操作粒度太死。以前的做法常把一次 LLM 回合視為一個標準單位，好像每個決策都應該花差不多的推理成本。但真實研究不是這樣。有時候你只是要改個超參數，有時候你需要追一個奇怪的錯誤，有時候你發現整個方向錯了，應該立刻停下。

AIRA² 用 ReAct agent 加上動態 action scoping，讓 Agent 可以自己決定這一步要花多少計算。簡單的事快做，複雜的事多想，懷疑路線不對就先 debug。這其實比「更聰明的模型」更重要，因為它讓系統有節奏感，不會把所有問題都用同一種昂貴姿勢處理。

## 數字背後透露的是什麼

在 MLE-bench-30 上，AIRA² 24 小時達到 71.8%，72 小時進一步拉到 76.0%。單看數字當然不錯，但我覺得更值得注意的是它在 72 小時後還在穩定上升。這代表它不是靠短期撞運氣，而是真的把長時搜索的退化問題壓下來了。

如果把三個設計放在一起看，邏輯其實很完整。非同步 worker pool 解決的是吞吐量，HCE 解決的是評估偏差，動態 action scoping 解決的是操作粒度。三者不是平行加分，而是互相補洞。少一個，系統都會重新露出舊毛病。

## 這篇論文真正值得帶走的判斷

我最在意的，不是它把 percentile 往上推了幾個點，而是它提醒我們一件很容易被忽略的事：Agent 的問題，很多時候不是 reasoning 不夠，而是系統設計還停留在單次互動的思路。

只要任務牽涉反覆試錯，評估機制就必須抗噪，排程就必須非同步，行動粒度就必須能伸能縮。這些聽起來不像 AGI 宣言，比較像基礎工程，可真正能讓研究 Agent 站穩的，往往就是這些地方。

如果你正在做 RAG、AutoML、批次實驗平台，AIRA² 不一定能整包搬走，但它至少給了一個很紮實的提醒：先把系統裡的等待、噪聲和僵硬流程處理掉，Agent 才有機會真的變成研究者，而不是一個看起來很忙的排程器。
