很多團隊在用外部 LLM API 時，都有一種很熟悉的無力感。

模型很強，但很貴。你知道它有能力，可是每多跑一次長推理、每多疊一層 agent，帳單就跟著發熱。於是大家開始想辦法省，有人裁 prompt，有人做快取，有人換便宜模型。AWS 這篇 ExecTune 論文選了另一條更有趣的路。它不直接跟黑盒大模型硬碰，而是在前面放一個很小的 guide model，專門負責指路。

乍看之下，這像是老派的 prompt engineering 升級版。但論文真正抓到的，不是「怎麼給出更聰明的建議」，而是「這個建議，大模型到底願不願意照做」。

## 為什麼很多好策略最後都死在執行

作者把整個系統叫做 Guide-Core Policy。guide 先生成策略，core 再根據這份策略完成任務。比如數學題裡，guide 會先寫出解題框架，再交給黑盒模型推導細節。直覺上，這很合理。小模型負責規劃，大模型負責算力最重的部分。

問題在於，策略不是寫出來就算數。

如果 guide 給的建議太抽象、太冗、太不合 core 的習慣，黑盒模型未必能執行。它可能忽略、誤讀，或者乾脆照自己的路走。於是你得到一個很尷尬的系統，前面看起來很聰明，後面卻沒有真的被帶動。

論文把這件事命名為 executability，也就是可執行性。我覺得這個詞抓得很準。很多提示工程失敗，不是因為想法差，而是因為目標模型根本消化不了。

## ExecTune 真正調教的是哪一段

不是 core，而是 guide。

這很重要，因為黑盒 API 最大的限制就是你不能碰它的權重。既然不能改 core，那就只能把 guide 訓練到更懂 core。ExecTune 分兩階段做這件事。

第一階段很像篩資料。先讓強 teacher model 為每道題目產生策略，再把「題目加策略」丟給目標 core 實際作答。只有那些真的幫 core 做對題目的策略，才留下來當 SFT 訓練資料。這一步很樸素，但也很致命。因為它把訓練集從「看起來不錯的策略」換成「已被證明能執行的策略」。

第二階段再加上 structure-aware GRPO。除了看答案對不對，還看策略格式有沒有固定、會不會偷洩漏答案、加了策略後是否反而讓 core 退步。這等於在逼 guide 不只會出主意，還要出一份能被接球的主意。

## 最驚人的地方不是分數，而是角色反轉

論文裡最醒目的結果，是用 Qwen3-1.7B 當 guide，搭配 Claude 3.5 Haiku 當 core，最後在 GSM8K 上把 Haiku 單跑的 70.89% 拉到 93.56%。這不只是小幅優化，而是直接改變你看待系統組裝的方式。

以前我們比較容易用模型排行思考產品。誰比較強，就買誰。誰比較貴，就少用點。ExecTune 暗示另一種思路：也許真正重要的不是買到更大顆的腦，而是替現有的大腦找到一個會下指令的前導。

這件事對成本敏感的團隊特別有吸引力。因為你不是每次都調更貴的核心模型，而是先把便宜的小模型訓練好，讓它在前面把路鋪平。整體能力升上去，推論單價卻不一定一起爆炸。

## 這套方法的限制藏在哪裡

第一，它現在主要驗證的是單輪、單策略場景。現實系統常常不是一次交代完，而是在工具呼叫、反思、修正中來回走。guide 如果要在多輪互動裡持續有效，問題會複雜很多。

第二，這套做法仍然依賴一個更強的 teacher model 來幫忙收集可執行策略。也就是說，成本沒有消失，而是從線上推論挪到離線建資料的那一頭。這很合理，但得算清楚帳。

## 我會怎麼讀這篇論文

如果只看表面，ExecTune 像是在講一種新的黑盒 prompt 優化法。但我覺得它更像是在提醒大家，設計 LLM 系統時，真正的界面不是文字本身，而是文字能不能在另一個模型裡引發穩定的行動。

這個角度很值得保留。因為只要你的系統裡有多模型協作，有 planner、有 executor、有 judge，就一定會遇到同樣的問題。最好的指令，不是最聰明的那一句，而是最能讓下一個模組真的照著做的那一句。

從這個角度看，ExecTune 不只是 guide model 的訓練技巧，它比較像是一種工程紀律。少迷信漂亮策略，多檢查策略有沒有落地。
