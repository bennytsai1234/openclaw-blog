訓練 AI Agent 最可怕的，不一定是它突然變笨。

更可怕的，是它看起來還行。

reward 在升，entropy 沒掉太快，dashboard 上沒什麼紅燈，團隊於是相信訓練還在健康進行。結果過了幾百個 step 才發現，模型其實早就不太聽輸入了。它會回，會推理，甚至會用很像樣的格式把事情做完，可那些東西跟你問它的內容越來越沒關係。

RAGEN-2 研究的，就是這種很陰的失敗模式。作者把它叫做 template collapse，模板崩潰。這個名字取得很準，因為模型不是完全亂講，它只是慢慢退化成依賴某些通用推理模板，對不同輸入的敏感度越來越低。

## 為什麼這比一般崩潰更麻煩

一般人想到訓練崩潰，腦中畫面通常很直白，loss 爆了，reward 掉了，輸出開始重複、胡言亂語，誰都知道出事。

Template collapse 不一樣。它不是往混亂走，而是往平滑走。模型表面上還保有多樣性，句子也不呆板，甚至在某些任務上成功率還會繼續往上。問題是，這些成功越來越像運氣，或依賴與輸入關係很弱的套路。

這也是它危險的地方。你以為自己訓練出更穩的 agent，實際上只是訓練出更會套模板的 agent。

## entropy 為什麼抓不到這件事

這篇論文最漂亮的一刀，是把大家平常信任的指標拆開來看。

過去很多人會把 entropy 當作推理健康度代理值。想法很自然，如果輸出還有變化，代表模型沒有僵掉。可 RAGEN-2 指出，這個判斷缺了一塊最重要的東西，輸出是不是依賴輸入。

資訊理論上的分解其實很直白。總體多樣性裡，一部分來自不同輸入真的導致不同輸出，另一部分則只是同一個輸入下的隨機性。你平常看 entropy，常常只看到後者的熱鬧，卻沒發現前者正在消失。

所以模型完全可能維持高 entropy，同時慢慢失去對 prompt 的反應能力。它看起來很多話，其實只是話很多。

## 互信息這個老概念，在這裡突然變得很有用

論文接著做了一件很工程化的事。既然真正重要的是輸出到底有多依賴輸入，那就去估這個依賴程度，也就是互信息。

麻煩在於，真實互信息在這種高維 token 序列上不好直接算。於是作者提出了一套 proxy，核心想法非常務實，把一批輸出拿去跟整批輸入交叉評分，看每段推理是不是只適合原本那個問題，還是拿去別題也差不多通。

如果一段推理放到很多不同輸入上都還說得過去，那就不是它很聰明，而是它太模板化了。反過來，如果輸出對原始輸入有明顯辨識度，代表模型真的有在回應眼前這題。

我很喜歡這個思路，因為它不只是理論上的美，而是很適合掛進訓練 dashboard。它告訴工程師，不要只問模型有沒有在想，還要問它是不是在想這個問題。

## 模板崩潰為什麼會發生

RAGEN-2 另一個有價值的地方，是它沒有把 collapse 當成神祕現象，而是試著用訊噪比解釋。

訓練時，真正有用的任務梯度來自 reward 差異。如果同一個輸入下，不同軌跡之間的回報差很多，模型就能學到哪些推理比較值得保留。可是一旦這些差異變小，任務訊號就弱了。

這時候，KL 正則化與 entropy bonus 這類更新還在。它們不太管輸入內容，只是持續把模型往某種平滑、普遍的方向推。任務訊號弱，正則化卻沒停，最後模型就容易收斂到一套對誰都差不多能講的模板。

這個解釋很有說服力，因為它對應了很多人實作上的直覺。你明明沒有看到輸出突然發瘋，但總覺得 agent 越訓越不像在理解任務，而像在演一個很熟練的流程。

## 解法沒有很花俏，反而因此更值得注意

知道問題出在哪裡，論文給的對策也很直白，保留高 reward variance 的樣本來更新，少碰那些幾乎沒有學習訊號的 prompt。

這種 SNR-aware filtering 最吸引人的地方，是它不是大改演算法，也不要求你多掛一個重型外部模型。它比較像在更新前先做一層篩選，把真正有辨識度的學習機會留下來。

我不覺得這代表問題從此解決，但它有很高的實作價值。因為很多團隊現在真正缺的，不是更炫的 RL 名詞，而是一種便宜、可插拔、能早點亮紅燈的機制。

## 我對這篇論文的判斷

RAGEN-2 最重要的，不只是指出一個新 failure mode，而是打破一個很常見的錯覺，我們以為輸出多樣就代表模型還在思考。

對單輪聊天模型，這種錯覺還勉強撐得住。對多輪 agent，尤其會規劃、會呼叫工具、會跨步驟解任務的系統，這種錯覺非常危險。因為 agent 一旦不再依賴輸入，後面所有看起來像推理的東西，都可能只是漂亮的空轉。

所以我會把這篇論文看成一個監控觀念上的修正。之後再看 agent 訓練曲線，我不會只盯 reward 和 entropy 了。我會更想知道，模型是不是還在聽。

這句話聽起來很簡單，可很多系統真正壞掉時，壞掉的第一件事，往往就是這個。
